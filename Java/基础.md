## 类型提升
### 一、基本规则（Java 类型提升）

在**算术表达式中**，类型提升遵循以下顺序和规则

| 类型                    | 升级为                          |
| --------------------- | ---------------------------- |
| `byte`、`short`、`char` | 都会提升为 `int`                  |
| `int`                 | 若和 `long` 运算，则提升为 `long`     |
| `long`                | 若和 `float` 运算，则提升为 `float`   |
| `float`               | 若和 `double` 运算，则提升为 `double` |
### 二、具体规则 + 示例

#### `byte`、`short`、`char` → 自动提升为 `int`
```
byte b1 = 10;
byte b2 = 20;
// byte b3 = b1 + b2; // ❌ 错误，结果是 int，不能自动赋值给 byte
int b3 = b1 + b2;     // ✅ 正确

char c1 = 'A';
int c2 = c1 + 1;      // ✅ 正确，'A' 是 65，c2 = 66

```

#### `int` + `long` → 结果为 `long`4444
```
int i = 1000;
long l = 2000L;
long result = i + l;  // ✅ i 被提升为 long，结果是 long

```

#### `float` + `double` → 结果为 `double`
```
float f = 3.14f;
double d = 2.71;
double result = f + d; // ✅ f 被提升为 double
```

### 三、常量表达式的特殊优化（窄化）
如果参与运算的是**常量字面量（constant expressions）**，结果在目标类型范围内，**Java 编译器允许窄化赋值**：
```
final byte b1 = 10;
final byte b2 = 20;
byte sum = b1 + b2;  // ✅ 编译器优化，允许，因为是常量，结果在 byte 范围内

byte x = 10;
byte y = 20;
// byte z = x + y;    // ❌ 错误，x+y是int，不能赋值给byte
```
### 四、复合赋值运算符的“隐藏转换”

如 `+=`、`-=` 等操作符，**会自动强转回原类型**：
```
byte b = 10;
b += 1;       // ✅ 隐式强转，相当于 b = (byte)(b + 1)
b = b + 1;    // ❌ 错误，需要手动强转
```

### 五、类型提升优先级图
```
byte, short, char
        ↓
        int
        ↓
       long
        ↓
      float
        ↓
     double
```
## 二进制的 原码 反码 补码

1. 二进制的最高位是符号位：0表示正数，1表示负数（老韩口诀：0->01- -）
2. 正数的原码，反码，补码都一样（三码合一）
3. 负数的反码=它的原码符号位不变，其它位取反（0->1,1->0）
4. 负数的补码=它的反码+1，负数的反码= 负数的补码- 1
5. 0的反码，补码都是O
6. java没有无符号数，换言之，java中的数都是有符号的
7. 在计算机运算的时候，都是以补码的方式来运算的.
8. 当我们看运算结果的时候，要看他的原码（重点）

| 名称     | 含义                     |
| ------ | ---------------------- |
| **原码** | 符号位 + 数值部分（最直观）        |
| **反码** | 负数 = 原码按位取反（除了符号位）     |
| **补码** | 负数 = 反码再 +1（计算机内部实际使用） |
#### 示例

| 编码方式   |     +5      |     -5      |
| ------ | :---------: | :---------: |
| **原码** | `0000 0101` | `1000 0101` |
| **反码** | `0000 0101` | `1111 1010` |
| **补码** | `0000 0101` | `1111 1011` |

1. ==**当调用对象方法的时候，该方法会和该对象的内存地址/运行类型绑定**==
2. 当调用对象属性时，没有动态绑定机制，哪里声明，那里使用


### DTO — Data Transfer Object（数据传输对象）

**用途：** 在不同层之间传输数据，尤其是跨服务、跨网络传输。

**场景：** 前端调用后端接口，或微服务之间互相调用。

java

```java
// 用户注册时，前端传给后端的数据
public class UserRegisterDTO {
    private String username;
    private String password;
    private String email;
    // 不包含 id、createTime 等数据库字段，因为还没入库
}
```

**特点：**

- 只带必要字段，不暴露数据库结构
- 可以做参数校验（加 `@NotNull` 等注解）
- 常见于 Controller 的入参

---

### VO — View Object（视图对象）

**用途：** 专门给前端展示用的对象，按页面需要组装数据。

**场景：** 用户查看个人主页，后端把用户信息 + 订单数量 + 积分组合成一个 VO 返回。

java

```java
// 个人主页展示用的数据
public class UserProfileVO {
    private String username;
    private String avatarUrl;
    private Integer orderCount;   // 来自订单表
    private Integer points;       // 来自积分表
    private String levelName;     // "黄金会员"（计算得出）
    // 密码、盐值等敏感字段绝对不出现
}
```

**特点：**

- 面向页面/接口响应，字段以"好展示"为标准
- 可能是多表聚合的结果
- 常见于 Controller 的返回值

---

### AO — Application Object（应用对象）

**用途：** Service 层内部使用的业务对象，承载业务逻辑处理过程中的中间数据。

**场景：** 下单时，Service 层需要同时处理商品信息、库存、优惠券，把这些组合在一起传递。

java

```java
// 下单业务处理时内部流转的对象
public class OrderCreateAO {
    private Long userId;
    private List<CartItem> items;
    private CouponInfo coupon;       // 优惠券信息
    private AddressInfo address;     // 收货地址
    private BigDecimal finalPrice;   // 计算后的最终价格
}
```

**特点：**

- 只在 Service 层内部流转，不对外暴露
- 解决一个方法参数太多的问题
- 在阿里巴巴规范中明确提出，但实际项目中用得相对较少

---

### 一张图看清区别

```
前端请求
    │
    ▼
Controller  ←── 接收 DTO（入参）
    │
    ▼
Service     ←── 内部用 AO（业务流转）
    │
    ▼
Repository  ←── 操作 Entity/PO（数据库实体）
    │
    ▼
Service     ──► 组装 VO
    │
    ▼
Controller  ──► 返回 VO（出参）给前端
```

---

### 总结对比

|DTO|VO|AO|
|---|---|---|---|
|含义|数据传输对象|视图对象|应用对象|
|方向|前端 → 后端（入参）|后端 → 前端（出参）|Service 层内部|
|关注点|接收参数、校验|展示数据、安全|业务逻辑流转|
|使用频率|★★★★★|★★★★★|★★★|


实际项目中 **DTO 和 VO 是最常用的**，很多小项目为了简单会把两者合并用，但随着业务复杂度上升，分开维护会让代码职责更清晰。=


# 计算机基础入门

> 从 Bit 到浮点数，一步一步理解

---

## 第一步：计算机只认识 0 和 1

计算机里所有东西，最底层只有两种状态：**0 和 1**。

可以想成电灯的开和关：

- 0 = 关
- 1 = 开

一个这样的「开关」，叫做一个 **Bit（位）**。

---

## 第二步：一个 Bit 能表示什么？

一个 Bit 只有两种可能：

```
0
1
```

只能区分 **2 种**状态，太少了。

---

## 第三步：多个 Bit 组合起来

**2 个 Bit：**

```
00
01
10
11
```

能表示 **4 种**状态（2² = 4）

**3 个 Bit：** 能表示 **8 种**状态（2³ = 8）

**规律：n 个 Bit → 能表示 2ⁿ 种状态**

---

## 第四步：8 个 Bit = 1 个 Byte（字节）

8 个 Bit 组合在一起，叫做 **1 Byte（字节）**：

```
0 0 0 0 0 0 0 0
```

能表示 **2⁸ = 256 种**状态。

---

## 第五步：怎么用这 256 种状态表示数字？

最简单的方式，直接对应：

```
00000000 → 0
00000001 → 1
00000010 → 2
00000011 → 3
...
11111111 → 255
```

这就是**无符号整数（uint8）**，范围 0 ～ 255。

---

## 第六步：有符号 vs 无符号整数

想表示负数怎么办？把**第一个 Bit 借出来**表示正负号：

- 第一位是 `0` → 正数
- 第一位是 `1` → 负数

剩下 7 个 Bit 表示数值，这就是**有符号整数（int8）**，范围 -128 ～ +127。

|有符号 int8|无符号 uint8|
|---|---|---|
|能表示负数|✅|❌|
|最小值|-128|0|
|最大值|+127|255|
|总共几个数|256 个|256 个|

总共能表示的数量一样，只是**范围不同**。

---

## 第七步：整数不够用——浮点数

比如 3.14、0.001 这种数字，用整数没法精确表示。所以发明了**浮点数**，用更多的 Bit，分成三段来存：

|段|作用|
|---|---|
|符号位|正数还是负数|
|指数位|小数点在哪里（数值范围大小）|
|尾数位|实际的数字内容（精度高低）|

---

## 第八步：不同位数的浮点格式

|名字|总Bit数|精度|数值范围|常见用途|
|---|---|---|---|---|
|FP32|32位|高|极大|AI 训练标准格式|
|FP16|16位|中|较小|省内存，但范围小|
|BF16|16位|中低|极大|AI 训练更稳定|
|INT8|8位|低（整数）|有限|AI 推理，最省内存|

位数越多 → 精度越高，但占的内存也越多。

---

## 总结

```
1 个 Bit            →  0 或 1
8 个 Bit（1 Byte）  →  256 种状态
全部表示正数         →  无符号整数（0 ~ 255）
留一位表示正负       →  有符号整数（-128 ~ 127）
表示小数             →  浮点数 FP32 / FP16 / BF16
```


# 数据类型与内存的关系

## 每种类型占多少内存？

|类型|位数|字节数|能表示的范围|
|---|---|---|---|
|INT8|8位|1 Byte|-128 ～ 127|
|INT32|32位|4 Byte|-21亿 ～ 21亿|
|FP16|16位|2 Byte|小数，范围较小|
|BF16|16位|2 Byte|小数，范围大|
|FP32|32位|4 Byte|小数，精度高|

**换算：** 8 Bit = 1 Byte，1024 Byte = 1 KB，1024 KB = 1 MB

---

## 为什么 AI 模型那么占内存？

一个大模型有几百亿个「参数（权重）」，每个参数就是一个数字。

以 700 亿参数的模型为例：

|格式|每个参数占用|总内存|
|---|---|---|
|FP32|4 Byte|**280 GB**|
|FP16|2 Byte|**140 GB**|
|INT8|1 Byte|**70 GB**|
|INT4|0.5 Byte|**35 GB**|

这就是为什么要做**量化**——把 FP32 压成 INT8，内存直接省 75%。

# 内存 vs 硬盘

## 一句话区别

- **内存（RAM）**：正在用的东西，放在这里，快但贵，断电消失
- **硬盘（SSD/HDD）**：存起来的东西，放在这里，慢但便宜，断电还在

---

## 生活类比

想象你在做饭：

||内存|硬盘|
|---|---|---|
|类比|🍳 灶台上的食材|🧊 冰箱里的存货|
|特点|随手可取，操作快|存得多，但要去拿|
|断电后|食材没了（丢失）|冰箱还在（保留）|

---

## 详细对比

||内存（RAM）|硬盘（SSD）|
|---|---|---|
|速度|极快（纳秒级）|较慢（微秒级）|
|容量|小（8～64 GB）|大（512 GB～几 TB）|
|价格|贵|便宜|
|断电|数据消失|数据保留|
|用途|运行中的程序|存文件、系统、软件|

---

## 和 AI 的关系

运行一个 AI 模型的流程：

```
硬盘（模型文件存在这里）
    ↓ 启动时加载
内存（模型权重读进来）
    ↓ 计算时使用
CPU / GPU（真正做运算）
```

所以你会看到：

- **「模型大小 70GB」** → 指硬盘上的文件大小
- **「需要 80GB 显存」** → 指运行时占用的内存（GPU 内存）

显存不够 → 模型根本跑不起来，不是跑得慢，是直接报错。
