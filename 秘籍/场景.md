#### 大文件分片上传 (Large File Chunked Upload)

**面试题：**

> “假设用户需要上传一个 5GB 的超大文献原稿。请描述你的完整前端上传方案，如何实现秒传、断点续传？并说明如何与后端 API（如 Python）配合？”

**考察点：** File API (`slice`)、Hash 计算、并发控制。

**参考答案：**

1. **文件分片 (Chunking)：** 利用 `File` 对象继承自 `Blob` 的 `slice` 方法，将 5GB 的大文件按固定大小（如 5MB）切割成 `ArrayBuffer` 切片数组。
    
2. **计算 Hash (秒传的核心)：** 不能直接将 5GB 文件读入内存计算。需要开启一个 **Web Worker**，利用 `spark-md5` 库一段一段地读取分片字节流，增量计算出整个文件的唯一 Hash。
    
    - **秒传：** 上传前将 Hash 发给后端。如果 Python 后端数据库中已记录该 Hash，直接返回上传成功。
        
3. **并发控制与断点续传：** 通过 Promise 封装上传请求。由于浏览器并发限制，我会实现一个请求并发控制器（维持最大 6 个并发）。如果某一切片上传失败，记录其索引并重试。每次上传前，可以向后端查询该文件已上传的切片索引列表，前端据此过滤掉已成功的分片，实现断点续传。
    
4. **后端合并请求：** 切片全部发送完毕后，前端发一个 `merge` 信号给后端。Python 后端根据接收到的文件名、Hash 和分片总数，在服务器临时目录中按顺序合并切片，并校验最终文件 Hash 是否一致。

####  虚拟滚动 (Virtual Scrolling)

**面试题：**

> “如果列表中的每一项高度是**动态未知**的（比如有的日志摘要很长，有的只有一行），你该如何在 Vue 中实现动态高度的虚拟滚动？”

**考察点：** DOM 布局原理、`nextTick`、ResizeObserver、位置缓存。

**参考答案：** 动态高度的虚拟滚动核心思路是**“先预估，后修正”**。

1. **初始化预估：** 先设定一个“预估高度”（如 `50px`）。基于预估高度，计算出占位区域的总高度，并通过 `computed` 截取当前可视区域需要渲染的数据（`startIndex` 到 `endIndex`）。
    
2. **渲染并获取真实高度 (Vue 生命周期介入)：** 将截取的数据渲染为 DOM。在这里我会利用 Vue 的 `nextTick`（确保 DOM 已更新）或者对列表项节点绑定 `ResizeObserver`，来获取这些刚渲染出来的元素的真实高度（`offsetHeight`）。
    
3. **更新缓存与位置修正：** 在组件内部维护一个响应式的 `positions` 数组缓存，记录每一项的 `top`、`bottom` 和真实的 `height`。当获取到真实高度后，更新对应的缓存记录，并同步修正后续所有元素的 `top` 和 `bottom` 值。
    
4. **滚动偏移量处理：** 监听 `scroll` 事件时，通过二分查找 `positions` 缓存，快速定位当前的 `startIndex`。同时，利用 Vue 的 `style` 绑定（`:style="{ transform: 'translateY(' + offset + 'px)' }"`）动态调整可视区域列表的偏移位置。
##### 核心原理概述

虚拟滚动的本质是：**只渲染可视区域及其缓冲区内的 DOM 节点，通过滚动事件实时计算并更新可视范围内的 DOM 数据，从而保持 DOM 节点数恒定。**

---

###### 一、 定高虚拟滚动：纯数学计算

如果每一项高度固定，我们可以利用简单的**数学公式**实现 O(1) 复杂度的定位，这是最基础的方案。

###### 实现原理

1. **数据状态：** 维护 `startIndex` (起始索引) 和 `endIndex` (结束索引)。
    
2. **公式推导：**
    
    - 已知：`containerHeight` (可视区高度), `itemHeight` (固定单项高度), `scrollTop` (滚动偏移量)。
        
    - 计算起始索引：$startIndex = \lfloor scrollTop / itemHeight \rfloor$
        
    - 计算结束索引：endIndex = startIndex + \lfloor containerHeight / itemHeight \rfloor + 1
        
3. **渲染偏移：**
    
    - 列表容器需要设置 `padding-top` 或 `transform: translateY`，将渲染出的 DOM 块“推”到正确的位置，让用户感觉自己在滚动一个巨大的列表。
        

**面试回答技巧：**

> “定高虚拟滚动的核心是**由 scrollTop 直接映射索引**。由于高度固定，我们不需要维护额外的数据结构，计算复杂度为 O(1)，性能损耗极低，是最高效的实现方式。”

---

###### 二、 不定高虚拟滚动：缓存与修正

当每一项高度不确定时，上述公式失效了，因为我们无法预知任意索引处的 `scrollTop` 对应哪一项。这时候需要**维护一个位置缓存表**。

###### 实现原理

1. **预估高度 (Estimated Height)：**
    
    在数据初始化时，给每一项分配一个预估高度（如 50px）。
    
2. **缓存表 (Position Cache)：**
    
    维护一个数组 `positions`，记录每一项的 `top`、`bottom` 和 `height`。
    
    JavaScript
    
    ```
    // 示例结构
    [
      { index: 0, top: 0, bottom: 50, height: 50 },
      { index: 1, top: 50, bottom: 120, height: 70 }, // 真实高度渲染后更新
      ...
    ]
    ```
    
3. **核心步骤 (Render-Measure-Update)：**
    
    - **渲染：** 根据当前 `startIndex` 渲染可视区的 DOM。
        
    - **测量：** 利用 `ResizeObserver` 或生命周期钩子（如 Vue 的 `nextTick` + `getBoundingClientRect`）获取 DOM 的真实高度。
        
    - **修正：** 获取真实高度后，更新 `positions` 数组中该项的数据，并触发一次“连锁反应”——因为当前项高度变了，后面所有项的 `top` 和 `bottom` 都需要重新计算。
        
4. **查找算法：**
    
    由于 `positions` 是有序的，查找当前 `scrollTop` 对应的 `startIndex` 时，不再使用简单的除法，而是使用**二分查找 (Binary Search)**，将时间复杂度控制在 $O(\log n)$。
    

**面试回答技巧：**

> “不定高虚拟滚动的关键在于**‘先预估，后修正’**。通过维护一个 `positions` 缓存数组来记录真实 DOM 高度，并将滚动位置的搜索优化为二分查找。如果遇到极端情况（比如高度变化剧烈），我会配合 `ResizeObserver` 实现自动更新，确保滚动条高度（`totalHeight`）计算的精确性。”

#### 大屏编辑器撤销重做

**面试官问法**
> 大屏编辑器的撤销重做是怎么实现的？

**口述回答**

撤销重做的核心是维护一个操作历史栈，记录每次状态变更。

我们用的是命令模式，把每一个用户操作封装成一个命令对象，命令对象包含两个方法，execute 执行操作，undo
撤销操作。每次用户做了操作，就把命令对象 push 到历史栈里。

撤销的时候从栈顶取出最近一条命令，调用它的 undo 方法恢复到上一个状态，同时把这条命令移到重做栈里。重做的时候从重做栈取出命令，重新执行
execute，再放回历史栈。

状态存储这块我们用的是快照模式，每次操作后把当前画布的完整 JSON 数据序列化存一份，撤销时直接恢复对应的快照，实现简单可靠。

**加分补充**

快照模式的问题是内存占用，如果用户操作很频繁，历史记录很多，每条都存完整快照会占用大量内存。我们的优化是限制历史栈的最大长度，比如最多保存
50 步，超过了就把最老的记录丢掉。

对于体积很大的画布，还可以改用差量存储，每次只记录变更的部分而不是完整快照，但实现复杂度会高很多，我们当前场景快照模式已经够用了。

#### AntV X6 节点和边的数据结构

**面试官问法**
> AntV X6 拓扑编辑器，节点和边的数据结构是怎么设计的？

**口述回答**

我们的拓扑编辑器是用来配置边缘设备之间的连接关系，最终要把配置下发到嵌入式运行环境，所以数据结构设计的核心原则是语义清晰、可序列化、便于下发。

节点代表一台设备，数据结构上除了 X6 本身需要的 id、位置、尺寸之外，我们在 data 字段里挂了业务数据，包括设备类型、设备
ID、设备名称、当前状态这些字段。

边代表设备之间的连接关系，除了 source 和 target 标识连接的两端节点之外，data 里挂了连接类型、通信协议、数据流方向这些业务字段。

整个画布序列化成 JSON 的时候，我们把节点数组和边数组分开存，节点里只保留业务相关的字段，X6
的渲染属性不做持久化，下发到嵌入式环境的是纯业务数据，渲染层和数据层彻底分离。

**加分补充**

这里有一个设计决策值得说——我们通过 JSON Schema 做了一层映射，Web 端的图形配置和嵌入式端的数据格式不完全一样，中间有一个转换层。这样
Web 端可以随意调整 UI 表达方式，不影响下发格式的稳定性，嵌入式那边也不需要关心 Web
端怎么画图，只关心收到的数据结构，解耦做得比较干净。

#### 遇到过最难的 Bug

**面试官问法**
> 遇到过最难的 Bug 是什么？怎么定位解决的？

**口述回答**

印象最深的是高频数据推送场景下的一个内存泄漏问题。

现象是页面运行一段时间后内存持续上涨，最终导致页面卡死甚至崩溃，但功能上看起来完全正常，复现也需要等比较长的时间。

定位过程是用 Chrome DevTools 的 Memory 面板，隔一段时间拍一次堆内存快照，对比两次快照找增量对象。发现有大量的
Vue 组件实例和 ECharts 图表实例没有被释放，数量随时间线性增长。

追查下去发现是两个原因。第一个是 MQTT 订阅的回调函数里持有了组件实例的引用，组件销毁时没有取消订阅，导致回调函数一直存在，组件实例无法被垃圾回收。第二个是
ECharts 实例在组件销毁时没有调用 dispose 方法，图表实例一直挂在内存里。

解决方案是在 onUnmounted 钩子里统一做清理，取消所有 MQTT 订阅，dispose 所有图表实例，之后内存稳定了，不再持续上涨。

**加分补充**

这个 Bug 给我的启发是，高频数据场景下的内存管理要特别谨慎，每一个订阅、每一个定时器、每一个第三方库实例，都要有对应的销毁逻辑。我们后来封装了一个
useCleanup 的组合函数，在里面统一注册需要清理的资源，组件销毁时自动执行，避免遗漏。

##### 3.http和https的区别
HTTP（超文本传输协议）是明文传输协议，信息在传输过程中极易被窃听、篡改或冒充；而 **HTTPS（HyperText Transfer Protocol Secure）** 是在 HTTP 的基础上引入了 **SSL/TLS 协议层**，实现了数据的加密传输、身份认证和数据完整性校验，是 HTTP 的安全升级版。

#####  interface 和type的区别
`interface` 专门用于定义对象结构，且支持“声明合并”；而 `type` 是类型别名，功能更广泛，支持联合类型、交叉类型、原始类型别名等。
