# AI 与编程范式 (AI & Trends)

## 核心概念：Fine-tuning vs RAG
在构建大语言模型（LLM）应用时，**Fine-tuning（微调）** 和 **RAG（检索增强生成）** 是优化模型表现的两大核心路径。
- **Fine-tuning**：让模型“**学**”到新知识或新习惯（改变大脑）。
- **RAG**：给模型“**看**”参考资料来回答问题（外挂书包）。
### 1. RAG (Retrieval-Augmented Generation)
RAG 结合了检索系统（如向量数据库）和生成模型。

- **流程：** 用户提问 $\rightarrow$ 系统在数据库中检索相关片段 $\rightarrow$ 将片段作为“上下文”连同问题喂给 LLM $\rightarrow$ LLM 生成答案。
- **✅ 优点：**
    - **即时性/动态性：** 只要更新数据库（知识库），模型就能回答最新的问题（例如“今天的新闻”）。
    - **准确性/可溯源：** 模型基于检索到的事实回答，减少幻觉，且可以给出引用来源。
    - **数据隐私：** 企业私有数据不需要被模型“记住”，可以通过权限控制检索范围。
    - **成本较低：** 不需要昂贵的训练算力，只需维护向量数据库。
- **❌ 缺点：**
    - **受限于 Context Window：** 输入的上下文长度有限，如果检索内容过多，可能丢失信息。
    - **依赖检索质量：** 如果检索到的内容是错误的或不相关的（Garbage In），回答也会是错误的（Garbage Out）。
    - **延迟：** 多了一个检索步骤，响应速度可能略慢。
### 2. Fine-tuning (微调)
- **✅ 优点：**
    - **风格一致性：** 非常擅长模仿特定的语气（如“客服口吻”、“鲁迅风格”）或特定的输出格式（如严格的 JSON 结构）。
    - **深度理解：** 对特定领域的术语、缩写或复杂逻辑有更好的理解。
    - **降低延迟/成本：** 微调后不需要塞入大量上下文，Prompt 更短，推理速度更快，Token 消耗更少。
- **❌ 缺点：**
    - **幻觉问题：** 仍然可能一本正经地胡说八道，它很难精确记住大量事实性细节。
    - **知识静态截止：** 模型训练完那一刻，知识就固定了。要更新知识，必须重新微调（昂贵且慢）。
    - **灾难性遗忘：** 过度微调可能导致模型丧失原有的通用能力。

### Loader 与 Plugin 的区别
1.  **Loader**
    -   **作用：** 用于处理模块中的资源文件，将它们转换为 Webpack 可以理解的模块。
    -   **资源处理：** 处理 JavaScript、CSS、图片、字体等，执行加载、转换、编译。
    -   **模块级别：** 工作在模块级别，通常处理单个文件，直接与模块内容交互。
    -   **配置：** 通过 `module.rules` 配置。
    -   **示例：** `babel-loader` (ES6+ 转 ES5), `css-loader`。

2.  **Plugin**
    -   **作用：** 扩展 Webpack 功能，执行自定义构建任务和优化。
    -   **构建过程控制：** 介入 Webpack 构建的生命周期（如代码压缩、文件生成、HTML 注入）。
    -   **应用级别：** 工作在应用程序级别，操作整个构建过程。
    -   **配置：** 通过 `plugins` 配置。
    -   **示例：** `HtmlWebpackPlugin` (生成 HTML), `UglifyJSPlugin` (代码压缩)。

# WebSocket 场景

### 1. 弱网环境下的稳定性与心跳机制
-   **智能重连机制 (Reconnection Strategy):**
    -   **指数退避 (Exponential Backoff):** 重连间隔递增 (1s -> 2s -> 4s...)。
    -   **随机抖动 (Jitter):** 增加随机数，避免拥塞。
    -   **限制重试:** 避免无限空耗。
-   **可靠消息保证 (Message Reliability):**
    -   **消息队列:** 前端维护发送缓冲队列 (Buffer Queue)。
    -   **ACK + 序列号:** 消息带 `id`/`seq`，收到 Server ACK 后才移除。
    -   **重连补发:** 连接恢复后检查队列，补发未 ACK 消息。

### 2. 高并发消息处理（聊天室 200 -> 500 人）
-   **逻辑层：** Set 防重、消息缓冲池。
-   **渲染层：** 虚拟列表 (Virtual List)、`requestAnimationFrame` 分批渲染/节流刷帧。
-   **视觉层：** 同用户消息合并展示。

### 3. 多 Tab 共享连接
-   **SharedWorker (最佳实践):** 所有 Tab 连接同一个 Worker，由 Worker 维持唯一的 WS 连接并分发消息。
-   **BroadcastChannel:** 主 Tab 维护连接，广播给其他 Tab。

### 4. WebSocket 消息顺序如何保证？
-   **后端：** 发送严格递增的 `seqId`。
-   **前端：** 维护 `nextExpectedSeqId` 和 **排序缓冲池**。
    1.  **命中:** `seqId === nextExpectedSeqId` $\rightarrow$ 渲染，`next++`。
    2.  **超前:** `seqId > nextExpectedSeqId` $\rightarrow$ 放入缓冲池暂存，开启定时器尝试“补洞”（应对丢包）。
    3.  **过期:** `seqId < nextExpectedSeqId` $\rightarrow$ 丢弃。

# Web Worker 相关

-   **适用场景：** 复杂 JSON 解析、路径规划、图像滤镜等 CPU 密集型任务。
-   **通信与性能：**
    -   **`postMessage` (结构化克隆):** 默认行为，数据拷贝，有性能开销 ($O(n)$)。
    -   **`Transferable` (零拷贝):** 适用于 `ArrayBuffer` 等二进制数据。
        -   **特点：** 瞬间完成 ($O(1)$)，但原对象在发送方会失效 (Neutering)。
        -   **注意：** 只能转移 `buffer` (如 `myUint8Array.buffer`)。
