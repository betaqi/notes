1. Loader：
	1.作用：Loader用于处理模块中的资源文件，将它们转换为Webpack可以理解的模块。
	2.资源处理：Loader处理各种资源文件，如JavaScript、CSS、图片、字体等，执行加载、转换、编译等任务。
	3.模块级别：Loader工作在模块级别，通常用于处理单个文件或模块，它们直接与模块的内用容交互。
	4.配置：Loader通过 module.rules进行配置，规定了哪些文件需要使用哪些Loader进行处理。示例：Babel Loader用于将ES6+ JavaScript转换为ES5，865CSS Loader用于加载和处理CSS文件等。
2. Plugin：
	1.作用：Plugin用于扩展Webpack的功能，执行各种自定义构建任务和优化。
	2. 构建过程控制：Plugin可以介入Webpack的构建过程，在不同的生命周期阶段执行任务，如代码压缩、文件生成、HTML注入等。
	3.应用级别：Plugin工作在应用程序级别，可以操作整个构建过程，包括资源文件的加载和输出。
	4.配置：Plugin通过plugins 进行配置，开发者可以根据需要添加不同的插件来扩展构建功能。
	3. 示例：HtmlWebpackPlugin用于生成HTML文件，UglifyJSPlugin用于代码压缩等。

- 缓存策略：理解强缓存 (`Cache-Control`, `Expires`) 与协商缓存 (`ETag`, `Last-Modified`) 的区别与流程 — 包括 304 请求是否会发送给服务器
	1. **强缓存** 浏览器在加载资源时，先看强缓存。
		`Cache-Control` (HTTP/1.1 - 主流)
		- **原理**：服务器返回一个**相对时间**（例如：`max-age=3600`，意思是“给你之后，3600秒内都是新鲜的”）。
		- **优势**：不依赖具体时间点，只算时间差，解决了 `Expires` 的问题。
		- **常见指令**：
		    - `max-age=xxx`：强缓存有效期。
		    - `no-cache`：**注意！** 这个名字很坑。它**不是**“不缓存”，而是**“不走强缓存，直接去走协商缓存”**（每次都去问服务器）。
		    - `no-store`：彻底不缓存，连协商缓存都不做（比如银行数据）。
	2. **协商缓存**当强缓存失效（过期了），或者被设置为 `no-cache` 时，浏览器就会向服务器发起请求，进行“协商”。
		1. `Last-Modified` 和 `If-Modified-Since` (基于时间)
			- **流程**：
			    1. 服务器给文件时，在 Header 里带上 `Last-Modified: 2024-01-01 12:00:00`。
			    2. 下次浏览器请求时，Header 里带上 `If-Modified-Since: 2024-01-01 12:00:00`。
			    3. 服务器对比时间：
			        - 时间一致 -> 返回 **304**（没改，你接着用旧的）。
			        - 文件更新了 -> 返回 **200** + 新文件 + 新的 `Last-Modified`。
			- **缺点**：只能精确到秒。如果一秒内修改了多次，它感知不到；或者文件只是被 `touch` 了一下，内容没变，它也会以为变了。
		2. `ETag` 和 `If-None-Match` (基于内容指纹 - 更准)
			- **流程**：
			    1. 服务器根据文件内容算出一个哈希值（指纹），放在 `ETag: "abc-123"` 里给浏览器。
			    2. 下次浏览器请求时，带上 `If-None-Match: "abc-123"`。
			    3. 服务器重新算文件指纹进行对比：
			        - 指纹一样 -> 返回 **304**。
			        - 指纹变了 -> 返回 **200** + 新文件 + 新的 `ETag`。

# WebSocket 场景

- WebSocket 连接在弱网环境下不稳定如何处理？心跳机制如何设计？
    1. **智能重连机制** (Reconnection Strategy)
		- **指数退避算法 (Exponential Backoff):** 重连间隔应该逐步递增。例如：1s -> 2s -> 4s -> 8s -> 30s(封顶)。
	    - **随机抖动 (Jitter):** 在退避时间上增加随机数，避免大量客户端在同一时刻发起重连。
		- **最大重试次数:** 设置上限，超过后提示用户手动刷新或检查网络，避免无限空耗电量。
	2. **可靠消息保证** (Message Reliability)
		WebSocket 也是基于 TCP 的，虽然 TCP 保证有序和不丢包，但在应用层连接断开重连的瞬间，数据可能会丢失。
		- **消息队列 (Message Queue):** 在前端维护一个发送缓冲队列。当 `readyState` 不为 `OPEN` 时，将消息压入队列。
		- **ACK 机制与序列号 (Sequence ID):**
		    - 每条消息带上唯一 `id` 或递增 `seq`。
	    - 发送后暂存，直到收到 Server 的 ACK 回执才从队列移除。
	    - **重连补发:** 连接恢复后，检查队列中未 ACK 的消息进行补发。
- 如果聊天室人数从 200 涨到 500，订阅和发布的消息过多，前端应该如何做消息去重、合并、批处理？
    1. **逻辑层****防重 & 缓冲**`Set` 记录 ID；**Message Queue + requestAnimationFrame 分批渲染**
    2.  降低 DOM 压力 **虚拟列表 (Virtual List)**；同用户消息视觉合并
    3.  **消息缓冲池：** 建立一个 `bufferArray`，收到消息先 push 进去，不做任何渲染。
    4. **节流刷帧 (Throttle / rAF)：** 使用 `requestAnimationFrame` 或 `setInterval` 定时从缓冲池中取出数据进行渲染。
- WebSocket 在 Tab 多开时应该如何避免每个 Tab 都创建新连接造成服务器压力？（如 BroadcastChannel、SharedWorker）
     1. SharedWorker (最佳实践)
	     **SharedWorker** 是浏览器专门为“跨页面共享逻辑”设计的 API。它是一个运行在独立线程中的脚本，可以被同源的多个 Tab、Iframe 共享。
	     **原理：**
			1. 所有 Tab 连接到同一个 `SharedWorker` 脚本。
			2. **只有 SharedWorker 内部创建一个 WebSocket 连接**。
			3. WebSocket 收到消息后，SharedWorker 遍历所有连接进来的 `Port`（即 Tab），将消息分发出去。
- WebSocket 消息顺序如何保证？需要前端做哪些设计？
	1. 后端要在每条消息中必须包含一个**严格递增的整数 ID** 前端不能收到消息就渲染，必须引入一个中间层：**排序缓冲池**。维护一个变量 `nextExpectedSeqId`（下一个期望的序号）。
		1. **命中 (Hit):** 收到的消息 `seqId === nextExpectedSeqId`。
		    - **动作：** 直接渲染/存入 Store。
		    - **更新：** `nextExpectedSeqId++`。
		    - **检查缓冲池：** 看看缓冲池里有没有下一条（比如刚才收到了 103，现在处理完 102，立马把 103 也处理了）。
		2. **超前 (Gap/Future):** 收到的消息 `seqId > nextExpectedSeqId`。
		    - **场景：** 期望收到 102，结果先收到了 104。说明 102, 103 还在路上，或者丢包了。
		    - **动作：** **千万不要渲染！** 将 104 放入 `bufferHeap`（缓冲堆/Map）暂存。
		    - **启动定时器：** 如果一定时间内（如 500ms）中间缺的消息还没来，触发“补洞”逻辑（见第3点）。
		3. **过期 (Stale):** 收到的消息 `seqId < nextExpectedSeqId`。
		    - **场景：** 已经处理到 105 了，突然收到了 101。
		    - **动作：** 说明这是迟到的重复包，直接丢弃（因为之前肯定处理过或补拉过）。
