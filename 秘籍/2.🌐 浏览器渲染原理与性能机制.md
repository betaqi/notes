## 一、渲染流程总览

当浏览器的 **网络线程** 接收到 HTML 文档后，会创建一个“渲染任务”，并将其放入 **渲染主线程（Main Thread）** 的消息队列中。

在事件循环机制的调度下，主线程取出该任务并开始整个渲染流程。整个过程由多个阶段组成：

> **HTML 解析 → 样式计算 → 布局 → 分层(Layering) → 绘制(paint) → 分块 → 光栅化 → 合成绘制（Draw）**

每个阶段都有明确的输入和输出，上一个阶段的输出会成为下一个阶段的输入，从而形成一条完整、严密的“渲染流水线”。

---

## 二、HTML 解析阶段

渲染的第一步是 **解析 HTML**，生成 DOM 树。

解析时浏览器会：

- 遇到 CSS 文件（`<link>`） → **启动预解析线程**进行下载与解析；
- 遇到 JS 文件（`<script>`） → **暂停 HTML 解析**，等待 JS 下载并执行完毕。

### ⚙️ 为什么 CSS 不阻塞 HTML 解析？

因为 CSS 的下载和解析工作在 **预解析线程** 中进行，主线程不会等待它完成。

### ⚙️ 为什么 JS 会阻塞 HTML 解析？

因为 JS 可能会修改 DOM（例如通过 `document.createElement`），所以主线程必须暂停解析 HTML，直到 JS 执行结束。

解析完成后，浏览器得到：

- **DOM 树**：HTML 的结构表示；
- **CSSOM 树**：由外部样式、内部样式、内联样式和浏览器默认样式组成。

---

## 三、样式计算（Style Computation）

主线程会遍历 DOM 树的每个节点，为其计算最终的样式（即 **Computed Style**）。

在此阶段：

- 颜色名会转换为 RGB 值，如 `red → rgb(255, 0, 0)`；
- 相对单位（`em`, `%` 等）会转化为绝对像素值（`px`）。

这一步的输出是 **带有计算样式的 DOM 树**。

---

## 四、布局阶段（Layout）

布局阶段会遍历 DOM 树，为每个节点计算几何信息（位置与尺寸），生成 **布局树（Layout Tree）**。

注意：

- `display: none` 的节点不会出现在布局树中；
- 伪元素（如 `::before`, `::after`）会出现在布局树中；
- 匿名块盒、匿名行盒等结构也可能出现在布局树中。

最终结果是：每个可见节点都拥有确定的几何信息。

---

## 五、分层阶段（Layering）

主线程会根据特定规则将布局树分为多个图层（Layer）。

分层的目的：

> 层与层之间可以独立绘制与合成，从而避免整个页面重复渲染，提升性能。

### 🧩 可能触发独立图层的情况

- 使用了 `transform`, `opacity`, `filter`, `will-change`；
- 形成新的堆叠上下文（`z-index`）；
- 出现滚动条、视频、Canvas 等。

分层完成后，每一层都会独立进入下一步的绘制流程。

---

## 六、绘制阶段（Paint）

主线程为每个图层生成绘制指令（Paint Commands），描述该层的内容如何被画出（文字、边框、阴影、图片等）。

完成绘制指令后，主线程将结果交给 **合成线程（Compositor Thread）**。

---

## 七、分块与光栅化（Rasterization）

合成线程接手后，会对图层进行进一步处理：

1. **分块（Tiling）**  
   每个图层会被划分为多个小块（Tile），方便并行处理。
2. **光栅化（Rasterization）**  
   合成线程将块信息交给 **GPU 进程**，由 GPU 将矢量绘制信息转为位图（bitmap）。  
   GPU 会优先处理**视口区域**内的块，并使用多个线程加速。

结果：每个图层的每个块都被光栅化成了位图。

---

## 八、合成与绘制阶段（Compositing & Draw）

合成线程拿到所有图层的位图后，会生成一系列「合成指令（Quad）」。

每个 Quad 会标识：

- 哪个位图要显示；
- 在屏幕的哪个位置；
- 是否有旋转、缩放、透明度变化等变形。

最后：

- 合成线程将这些指令交给 GPU；
- GPU 执行系统调用，将最终图像绘制到屏幕。

> **关键点：** transform、opacity 等变换发生在合成阶段，与主线程无关。这正是它们高效的原因。

---

## 九、Reflow（回流）

**Reflow** 的本质是：  
> 重新计算布局树（Layout Tree）。

当某些操作影响了元素的几何信息（如尺寸、位置、字体大小等），浏览器需要重新计算布局。

为提升性能，浏览器会将多次修改合并为一次回流操作，等到 JS 执行结束后再统一计算（异步进行）。

### ⚠️ 注意：
设置布局属性不会立刻回流（如 `dom.width = xxx` ) 他是存储在内存中的会在某个时机进行重新计算布局树。**因为这个机制有时候读取dom的width不是最新的**
如果在 JS 执行中访问了布局属性（如 `clientWidth`  `getBoundingClientRect` `offsetHeight`, `scrollTop` 等），浏览器会 **强制同步回流**，以保证获取的是最新值。
例子
```
1.
const dom = document.getElementById('box');
// 只改样式，不触发回流
dom.style.width = '200px';
dom.style.height = '100px';
console.log(dom.style.width); // "200px"，只是内存值
console.log(dom.clientWidth); // 强制回流，拿到最新宽度
// 使用 getBoundingClientRect() 同样会触发回流
console.log(dom.getBoundingClientRect().width); // 最新值


2. **CSS 优先级覆盖**
- 如果元素有 class 或外部 CSS 文件中定义了 width，且使用 `!important` 或更高优先级覆盖内联样式：
div.mybox {
  width: 300px !important;
}
dom.style.width = '200px';
console.log(dom.style.width);      // "200px"（内联存储）
console.log(dom.getBoundingClientRect().width); // 300px（渲染结果）
- 内联 style 值存在，但最终渲染被 CSS 覆盖 → 视觉上宽度不一致。
```
---

## 十、Repaint（重绘）

**Repaint** 的本质是：  
> 重新生成绘制指令，更新页面的可见样式。

当元素的可见外观发生改变（如颜色、背景、阴影等），但不影响布局时，会触发 Repaint。

由于布局信息也属于可见样式，所以：

> **Reflow 一定会导致 Repaint**，但 Repaint 不一定会导致 Reflow。

---

## 十一、为什么 transform 效率高？

`transform` 的改变只会影响合成阶段（Compositing），不会触发布局（Reflow）或绘制（Repaint）。

因此：

- 不会重新计算布局树；
- 不会重新生成绘制指令；
- 直接由合成线程或 GPU 处理变换。

这意味着：

- 即使主线程被阻塞，`transform` 动画依然可以流畅执行；
- 它的性能远高于依赖主线程的 `left` / `width` 等动画。

---

## ✅ 总结

| 阶段 | 描述 | 是否依赖主线程 |
|------|------|----------------|
| HTML 解析 | 生成 DOM 树 | ✅ |
| 样式计算 | 生成 Computed Style | ✅ |
| 布局 | 生成布局树 | ✅ |
| 分层 | 生成 Layer 树 | ✅ |
| 绘制 | 生成 Paint 指令 | ✅ |
| 分块 | 分割为 Tile | ✅（合成线程执行） |
| 光栅化 | 转为位图 | ❌（GPU 执行） |
| 合成绘制 | 图层合成、transform、opacity 变换 | ❌（Compositor/GPU 执行） |

---

## 📚 小结

> - 浏览器渲染是一条多阶段的流水线。  
> - 主线程负责解析、计算与绘制指令生成。  
> - 合成线程与 GPU 负责最终画面呈现。  
> - transform、opacity 动画能在合成层独立运行，因此性能极高。
