
## 一、浏览器常见线程角色

| 线程类型 | 功能 | 是否影响主线程 |
|-----------|------|----------------|
| **JS 主线程** | 执行 JS、调度任务 | ✅ 是核心 |
| **渲染线程（Render Thread）** | 绘制页面、执行重绘重排 | ✅ 互斥（JS 执行时暂停渲染） |
| **定时器线程（Timer Thread）** | 管理 `setTimeout` / `setInterval` 定时器 | ❌ 不直接执行 JS |
| **网络线程（Network Thread）** | 处理 HTTP 请求、WebSocket、SSE 等 | ❌ 不直接执行 JS |
| **事件触发线程（Event Thread）** | 管理用户输入、DOM 事件 | ❌ 不直接执行 JS |
| **Web Worker 线程** | 可并行执行计算任务 | ❌ 独立于主线程 |

---

## 二、事件循环（Event Loop）

JS 的执行模型基于事件循环（Event Loop），任务类型分为：

- 🧩 **宏任务（MacroTask）**
- ⚡ **微任务（MicroTask）**

### 常见分类

| 类型 | 举例 | 优先级 |
|------|------|--------|
| **宏任务** | `setTimeout`、`setInterval`、I/O、渲染、事件回调 | ⬇️ 低 |
| **微任务** | `Promise.then`、`MutationObserver`、`queueMicrotask` | ⬆️ 高 |

### 📍执行顺序

1. 执行主线程脚本  
2. 执行所有微任务（Promise等）  
3. 如果空闲，则进行渲染  
4. 取出下一个宏任务  
5. 重复循环

---

## 三、渲染线程与主线程的关系

### 🧩 关键点

- JS 主线程与渲染线程 **互斥**
- 当 JS 执行时，浏览器 **无法渲染页面**
- 渲染操作通常发生在：
  - 宏任务之间
  - 所有微任务执行完毕之后

### 📈 结果表现

- 频繁的 DOM 操作或长时间执行的 JS 任务，会导致 **掉帧、卡顿**
- `requestAnimationFrame()` 比 `setTimeout()` 更适合动画  
  → 因为它会在 **下一帧渲染前** 被调用

---

## 四、`requestAnimationFrame()` 触发时机

浏览器渲染通常是这样的一个循环（简化版）：

> **事件处理 → JS 执行 → 样式计算 → 布局 → 绘制 → 合成 → 下一帧**

`requestAnimationFrame` 的回调会插在：
🧠 **“样式计算 → 布局” 之前**  
也就是浏览器准备渲染下一帧画面之前调用。

---

### 💡 小结

| 结论 | 说明 |
|------|------|
| JS 执行会阻塞渲染 | JS 和渲染线程互斥 |
| CSS `transform`、`opacity` 动画可异步执行 | 由合成线程控制，不依赖主线程 |
| `requestAnimationFrame` 更平滑 | 与刷新率同步，帧级触发 |
| Web Worker 可并行计算 | 适合离线复杂逻辑或大数据处理 |

---

> ✍️ **建议**：  
> 在执行密集任务（如循环或复杂计算）时，使用  
> - `requestIdleCallback()`  
> - 或将任务拆分到多个宏任务中  
> 避免主线程长期被阻塞导致动画卡顿。
