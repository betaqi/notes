
> 从下到上理解网络协议，搞清楚 ping、curl、代理、端口这些日常工具到底在哪一层工作。

---

## 一、网络分层模型

理解网络的核心：**数据是一层层封装、一层层拆开的**。

常用的是 **TCP/IP 四层模型**（OSI 七层是理论模型，实际用四层就够）：

```
应用层    HTTP / HTTPS / DNS / SSH / FTP
传输层    TCP / UDP
网络层    IP / ICMP
链路层    以太网 / WiFi
```

每层只管自己的事，下层为上层服务。

---

## 二、网络层（IP 和 ICMP）

### IP（Internet Protocol）

- 负责"寻址"：每台设备有 IP 地址，数据包靠 IP 地址送达
- IPv4 地址：`192.168.1.1`
- IPv6 地址：`2001:db8::1`

### ICMP（Internet Control Message Protocol）

- 不传业务数据，只传控制和诊断消息
- `ping` 用的就是 ICMP（发 Echo Request，等 Echo Reply）
- `traceroute` 也靠 ICMP（看每一跳路由）
- **ICMP 不是 TCP，所以代理软件一般不代理它**

---

## 三、传输层（TCP 和 UDP）

### TCP（Transmission Control Protocol）

- 面向连接，可靠传输
- 三次握手建立连接，四次挥手断开
- 保证数据按顺序、不丢、不重复
- 慢一点，但可靠
- 用途：HTTP、HTTPS、SSH、Git、数据库

### UDP（User Datagram Protocol）

- 无连接，不保证送达
- 速度快，开销小
- 丢了就丢了，不重传
- 用途：DNS 查询、视频通话、游戏、QUIC（HTTP/3）

### 端口（Port）

IP 找到机器，端口找到机器上的具体程序。范围 0–65535，常见的：

| 端口 | 协议 |
|------|------|
| 80   | HTTP |
| 443  | HTTPS |
| 22   | SSH |
| 53   | DNS |
| 3306 | MySQL |
| 6379 | Redis |

---

## 四、应用层

### HTTP / HTTPS

- 网页用的协议，建立在 TCP 上
- HTTPS = HTTP + TLS 加密
- `curl https://google.com` 走的就是这条

### DNS

- 域名解析：`google.com` → `142.250.x.x`
- 默认走 UDP 53 端口

### SSH

- 安全远程登录，TCP 22

### WebSocket

- 全双工通信，建立在 HTTP 之上

## 五、代理相关（重点）

### HTTP 代理

- 只代理 HTTP/HTTPS 流量
- 客户端发 `CONNECT host:port` 让代理建立隧道
- 端口示例：`http_proxy=http://127.0.0.1:10808`

### SOCKS 代理

- 工作在更底层，能代理任意 TCP（甚至 UDP，看版本）
- SOCKS5 比 SOCKS4 多了认证和 UDP 支持
- 端口示例：`all_proxy=socks5://127.0.0.1:10808`

### 关键认知：哪些工具走代理？

| 工具 / 协议 | 走代理吗 |
|------------|---------|
| `curl https://...` | ✅ 走 |
| `git clone https://...` | ✅ 走（如果配了 git proxy） |
| `npm install` | ✅ 走（如果配了 npm proxy） |
| `ping` | ❌ 不走（ICMP） |
| `traceroute` | ❌ 不走（ICMP/UDP） |
| `nslookup` | ❌ 默认不走（DNS/UDP） |

> 这就是为什么 `proxy-on` 之后还是 ping 不通 google：ping 根本没经过代理。

## 六、常用排查命令

### Windows / PowerShell

```powershell
# 查端口监听
netstat -an | findstr 10808

# 测 TCP 连通性（替代 telnet）
Test-NetConnection google.com -Port 443

# 看 DNS 解析
nslookup google.com

# 看路由
tracert google.com

# 真 curl（不是 PowerShell 别名）
curl.exe -I https://google.com
```

### Mac / Linux

```bash
# 查端口监听
lsof -i :10808

# 测 TCP 连通性
nc -zv google.com 443

# DNS
dig google.com

# 路由
traceroute google.com

# curl
curl -I https://google.com
```

## 七、扩展知识（按兴趣往下挖）

- **TLS/SSL** — HTTPS 的加密层，证书、握手、SNI
- **HTTP/2 vs HTTP/3** — HTTP/3 跑在 UDP 上的 QUIC
- **NAT** — 家用路由器为什么多个设备能共用一个公网 IP
- **防火墙** — 丢包还是拒绝，区别在哪
- **MTU / 分片** — 为什么有时候 ping 通但下载断流
- **抓包** — Wireshark、tcpdump，能看到每一层数据长啥样

## 八、推荐学习路径

按这个顺序学，效率最高：

1. 先搞明白 IP / TCP / UDP / 端口（80% 日常够用）
2. 再学 HTTP 协议本身（请求方法、状态码、Header）
3. 然后是 TLS/HTTPS 怎么握手的
4. 想深入就抓包看，最直观

入门书推荐：

- 《图解 HTTP》
- 《图解 TCP/IP》

几个晚上能翻完，比看视频快。

## 速查：为什么 ping 不通但 curl 能通？

- `ping` 走 ICMP，**不经过 HTTP/SOCKS 代理**
- Google 等大厂边缘节点会主动丢弃 ICMP 包
- 测代理是否生效，永远用 `curl.exe -I https://www.google.com`，不要用 ping
