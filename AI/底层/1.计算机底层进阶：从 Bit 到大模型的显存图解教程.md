## 第一章：计算机的“一根筋”思维与大模型的显存初算

### 1.1 计算机是个“一根筋”的灯泡人

不管现在的大模型（LLM）聊起天来多么像人类，把它的 CPU 和 GPU 放大到微观层面，里面其实只有无数个小小的“开关”。

在计算机里，所有的信息最底层只有两种状态：**0** 和 **1**。你可以完美地把它们类比为电灯的开关：

- **`0`** = 关（低电平，没电）
    
- **`1`** = 开（高电平，有电）
    

这一个只能表示 `0` 或 `1` 的最小电灯泡单位，就叫做一个 **Bit（位）**。

### 1.2 组合暗号的威力：从 1 到 $2^n$

一个小灯泡只能表示 2 种状态（0 或 1），能放的暗号太少了。想要表示更复杂的数字，就必须把灯泡**并排组合**起来。

我们来看看并排摆放灯泡时，暗号数量是怎样爆炸式增长的：

- **1 个 Bit（1 盏灯）：** `0`, `1` $\rightarrow$ 只能区分 **2** 种状态 ($2^1$)
    
- **2 个 Bit（2 盏灯）：** `00`, `01`, `10`, `11` $\rightarrow$ 能区分 **4** 种状态 ($2^2$)
    
- **3 个 Bit（3 盏灯）：** `000`, `001`, `010`, `011`, `100`, `101`, `110`, `111` $\rightarrow$ 能区分 **8** 种状态 ($2^3$)

问题：3盏灯为什么是8个状态
    

> **💡 核心规律：**
> 
> 每增加一盏灯，能表示的暗号数量就**翻一倍**。因此，$n$ 个 Bit 组合起来，就能表示 **$2^n$** 种状态。

### 1.3 工业标准的诞生：为什么要规定 8 个 Bit = 1 个 Byte（字节）？

在计算机发展早期，每个科学家定义的“一组灯泡有几盏”都不一样，有人用 4 位，有人用 6 位。后来为了统一标准（同时为了能刚好装下英文大小写字母、数字和常用标点符号），工业界制定了一个铁律：**把 8 个 Bit 捆绑在一起，叫做 1 个 Byte（字节）。**                  

只要提到 1 个 Byte，计算机就知道里面有 8 个连在一起的 0 或 1（例如 `01011010`）。根据前面的规律，1 个 Byte 一共能表示：

$$2^8 = 256 \text{ 种状态}$$

问题：这个和计算器早起编码格式不统一有关系吗后来是用的utf8

### 1.4 怎么用这 256 种状态表示“整数”？

有了 1 个 Byte 的 256 种状态，我们就可以开始在内存里“映射”真正的数字了。

#### 方式 A：全部拿来存正数（无符号整数 `uint8`）

让状态从零开始，一个对一个。

- `00000000` $\rightarrow$ 对应数字 `0`
    
- `00000001` $\rightarrow$ 对应数字 `1`
    
- ...
    
- `11111111` $\rightarrow$ 对应数字 `255`
    
    这就是 **`uint8`**，它的范围是 **0 ～ 255**，总共刚好 256 个数字。


#### 方式 B：抢走一盏灯当正负号（有符号整数 `int8`）

现实生活中我们需要负数。计算机科学家想了个办法：**把最左边的那盏灯（最高位）抢过来当“正负号指针”**。

- 最左边是 `0` $\rightarrow$ 代表这是个**正数**
    
- 最左边是 `1` $\rightarrow$ 代表这是个**负数**
    

因为拿走了一位做符号，剩下用来表示数值大小的灯泡就只剩 7 个了（$2^7 = 128$ 种状态）。经过精妙的数学设计（补码系统），**`int8`** 能表示的范围变成了 **-128 ～ +127**。

> **⚠️ 注意：** 无论怎么变，1 个 Byte 能表示的数字**总个数永远是 256 个**，它只是把坐标轴往左平移了。

### 1.5 大模型连线：静态显存公式的推导

大模型里的所谓“参数（Weights，权重）”**，在底层其实就是一个个排好队的数字。而所谓的格式（INT8, FP16），本质上就是**每一个参数分到了几个 Bit 的灯泡。

我们现在直接把刚才学到的 `Bit` 和 `Byte` 拿来算一笔大模型的静态权重账：

- **7B 模型** = 这个模型有 **70 亿个参数**（7 Billion）。
    
- 如果我们用 **INT8** 格式去存它，意味着**每个参数占用 8 个 Bit（1 个 Byte）**。
    

$$\text{总字节数} = 7,000,000,000 \text{ Bytes}$$

$$\text{换算成 GB} = 7,000,000,000 \div 1024 \div 1024 \div 1024 \approx \mathbf{6.52\text{ GB}}$$

如果是主流未量化的 **FP16**（半精度浮点数），每个参数要用 16 个 Bit（也就是 **2 个 Byte**）。那么显存占用直接翻倍：

$$6.52\text{ GB} \times 2 = \mathbf{13.04\text{ GB}}$$
问题：字节单位换算

> ### 🧠 课后思考与承上启下
> 
> **思考题：** 既然我们通过严谨的数学计算得出，一个 7B 模型的 FP16 纯权重只需要 **13.04 GB** 的空间，那为什么在实际部署时，一块 16G 显存的显卡（如 RTX 4080）跑 7B 模型只要上下文拉长就会直接爆显存（OOM）？多出来的显存被什么吞掉了？ 
> 
> **承上启下：** 我们算出了整数（INT8）的大模型体积，但大模型内部包含了无数的概率和权重矩阵，它们充斥着 `0.00135` 或 `-3.14159` 这样的小数。既然 `int8` 只能表示整数，计算机到底是用什么样的“灯泡组合”来精确或者模糊地表达现实世界中的复杂小数的？这就是第二章要解决的——浮点数的本质。

## 第二章：浮点数的精妙解密与 AI 时代的数字“减肥”

### 2.1 为什么不能用“定点数”？

想用二进制表示小数，最直观的办法是“一刀切”：比如一共有 8 位，前 4 位表示整数，后 4 位表示小数。这叫**定点数**。

但这种方式极度僵硬：如果你想表示一个极大的数（如大模型的梯度），整数位不够用；如果你想表示一个极小的数（如微小的权重变化），小数位又不够精准。

为了解决这个矛盾，计算机科学家模仿了人类的**科学计数法**。

人类写大数或小数时会写成：$\pm 3.14 \times 10^{-3}$。在这个公式里，数字的精度由 $3.14$ 决定，数字的大小范围由 $10^{-3}$ 决定。

### 2.2 拆解 IEEE 754 浮点数标准

计算机把这种科学计数法搬进了二进制，将一段有限的 Bit 拆成了**三段**来存：

|**段名称**|**作用**|**类比科学计数法**|
|---|---|---|
|**符号位 (Sign)**|决定正负|最前方的 $\pm$ 号|
|**指数位 (Exponent)**|决定小数点点在哪里（决定数字的**范围**大小）|$10^n$ 里的那个 $n$|
|**尾数位 (Fraction/Mantissa)**|决定实际的数字内容（决定数字的**精度**高低）|前面的主体数字 $3.14$|

### 2.3 FP32、FP16 与 BF16 的生死之抉

在 AI 芯片和显存的发展史中，这三段的比例分配引发了一场产业革命。我们来看 32 位和 16 位浮点数的二进制切片对比：

```
FP32 (32-bit): [S] [EEEEEEEE] [FFFFFFFFFFFFFFFFFFFFFFF] 
               (1位符号 | 8位指数 | 23位尾数)

FP16 (16-bit): [S] [EEEEE] [FFFFFFFFFF] 
               (1位符号 | 5位指数 | 10位尾数)

BF16 (16-bit): [S] [EEEEEEEE] [FFFFFFF] 
               (1位符号 | 8位指数 | 7位尾数)
```

#### FP32（单精度浮点数）

拥有 8 位指数和 23 位尾数。精度极高，范围极大。它是传统科学计算和 AI 训练的黄金标准，但缺点是太胖了，每个数字啃掉 **4 个 Byte**。

#### FP16（半精度浮点数）

为了减肥，砍掉了一半的 Bit。它的指数位只有 5 位。这就导致了一个致命缺点：**它最大只能表示 65504**。在大模型训练过程中，激活值和梯度经常会超过这个数字，一旦超过，数据就会变成 `NaN`（Not a Number，数据溢出崩溃）。

#### BF16（Brain Floating Point，大模型的天选格式）

谷歌在一颗神经网络芯片（TPU）中做了一个非常激进且天才的尝试：同样是 16 位，**BF16 强行保持和 FP32 完全相同的 8 位指数位，直接把尾数位砍到只剩 7 位**。

- **结果：** BF16 精度变差了（数字变得像马赛克一样模糊），但是它**能伸能屈**，能表示的最大范围和 FP32 一模一样！大模型由于自身具有很强的容错和泛化能力，它“不怕模糊，就怕数据溢出”。因此，BF16 成为了如今大模型训练和推理的绝对标配。
    

> ### 🧠 课后思考与承上启下
> 
> **思考题：** 既然 BF16 牺牲了精度（尾数变短）却能让大模型正常运转，这是否意味着大模型对每个数字的“精准度”要求其实没有我们想象的那么高？如果连 BF16 这种“模糊”的数字都能跑，我们能不能更进一步，连浮点数的结构都不要了，直接用 4 个 Bit 的整数（INT4）来存大模型？
> 
> **承上启下：** 浮点数的拆分让我们知道了数字在显存里如何维持范围和精度的平衡。但是，16 位的数字对于想要塞进手机、本地电脑的大模型来说还是太大了。下一章，我们将彻底揭开“量化技术”的神秘面纱，看看如何把一个布满浮点数的精密矩阵，硬生生压缩成 INT4，却不让大模型变成白痴。

## 第三章：量化（Quantization）的数学魔法与运行时动态显存

### 3.1 量化不是简单的“四舍五入”

如果把大模型里类似 `0.234`、`1.567` 这样的浮点数直接四舍五入成整数，由于误差的层层累加，神经网络在经过几百层矩阵乘法后，输出的结果会彻底沦为随机乱码（大模型变白痴）。

工业界使用的量化，本质上是一种**线性映射（Linear Mapping）**。

### 3.2 线性量化的数学本质

想象一段弹簧。量化的本质是找到大模型某一层权重矩阵里的**最大值（Max）**和**最小值（Min）**，然后拉开一根“数字弹簧”，把这个范围内的所有浮点数，等比例地映射到整数的格子里。

其核心公式为：

$$Q = \text{round}\left(\frac{X}{S}\right) + Z$$

其中：

- **$X$**：原始的浮点数（FP16/BF16）
    
- **$S$ (Scale，缩放因子)**：弹簧的拉伸比例，计算公式为 $\frac{\text{Max} - \text{Min}}{2^b - 1}$（$b$ 为目标位宽，如 INT8 是 8 位，即 255）
    
- **$Z$ (Zero Point，零点)**：偏移量，确保原始浮点数里的 `0.0` 能够完美对应到量化后的某个整数上。
    

通过这种方式，大模型的一组权重被压缩成了两个部分：**一个极其节省空间的整数矩阵（INT4/INT8） + 极个别记录缩放比例的浮点数（Scale 和 Zero Point）**。

### 3.3 计算时的反量化（Dequantization）

你需要明确一点：目前的英伟达 GPU 核心（Tensor Core）在进行矩阵乘法时，底层最擅长的依然是浮点数运算。

大模型被量化成 INT4 塞进显存后，当 token 输入进来、GPU 开始计算的瞬间，硬件会在显存内部的超高速缓存中，通过公式 $X = S \times (Q - Z)$ 把 INT4 **瞬间解压回浮点数**参与计算。

- **为什么要这么折腾？** 因为大模型的计算瓶颈往往不在于 GPU 算得不够快，而在于显存把数据传输给 GPU 的速度太慢（**访存墙 Block**）。把 INT4 传过去再动态解压，传输的数据量减小了 75%，整体速度反而暴涨。
    

### 3.4 动态显存的幕后黑手：为什么 13G 的权重需要 16G 显存？

现在我们可以回答第一章留下的悬念了。在运行大模型时，显存并不是只用来装模型权重的（那是静态显存）。显存内部被切分成了三大块：

$$\text{总显存} = \text{静态模型权重 (Model Weights)} + \text{KV Cache (上下文缓存)} + \text{动态计算上下文 (Activation \& CUDA Overhead)}$$

1. **静态模型权重**：7B FP16 就是死死的 **13.04 GB**。
    
2. **动态计算上下文**：PyTorch 和 CUDA 驱动本身运行、矩阵计算中间产生的临时梯度，会固定吃掉 1G ~ 2G 空间。
    
3. **KV Cache（关键黑洞）**：大模型在生成文本时，每多生成一个字，就需要把前面所有产生过的注意力机制数据（Key 和 Value 矩阵）全部死死缓存在显存里，否则下一个字就无法建立长文本记忆。
    
    - **动态增高**：随着你和 AI 聊天的上下文（Context Length）从 1K 拉长到 8K、32K，KV Cache 会呈现线性爆炸式增长。这就是为什么权重只有 13G，但上下文一长，16G 显存瞬间崩塌的原因。

## 第四章：硬件拓扑的宿命——为什么显存不够大模型绝对跑不起来？

### 4.1 漏斗形的计算机存储金字塔

在计算机世界里，有一个永恒的物理铁律：**速度越快的存储介质，价格越贵，容量越小。**

```
     / \       [ 寄存器 / GPU Cache ]  - 纳秒级，容量以 MB 计 (极贵)
    /   \     -----------------------
   /     \    [    显存 (VRAM)      ]  - 速度达 TB/s，容量以 GB 计 (AI 命脉)
  /       \   -----------------------
 /         \  [    内存 (RAM)       ]  - 速度数十 GB/s，容量以 GB 计 
/___________\ [  固态硬盘 (SSD/HDD)  ]  - 毫秒/微秒级，容量以 TB 计 (极便宜)
```

### 4.2 内存与硬盘的“做饭类比”

我们可以用一个非常形象的生活场景来对应计算机的各级存储：

|**硬件体系**|**物理特性**|**厨房做饭类比**|**断电后的结局**|
|---|---|---|---|
|**硬盘 (SSD/HDD)**|慢、便宜、容量巨大|**家里的地下室/冰箱**：存了100斤大米和冻肉。要用时得走过去拿（慢）。|稳固。断电了，冰箱里的货还在。|
|**内存/显存 (RAM/VRAM)**|极快、贵、容量有限|**灶台旁的切菜案板**：只能放下正在切的一颗白菜和两块肉。随手就能抓到（极快）。|彻底清空。断电或关机，案板上的东西全部倒掉。|
|**计算核心 (CPU/GPU)**|负责疯狂做运算|**正在颠勺的厨师**：两只手（核心）只能处理锅里的东西，必须高频从案板上拿菜。|N/A|

### 4.3 为什么内存不够系统只是变卡，而显存不够大模型直接报错（OOM）？

#### 1. 电脑内存（RAM）的退路：虚拟内存

当你的电脑打开了 100 个 Chrome 标签页导致内存爆满时，操作系统（Windows/Linux）会启动保底机制：在硬盘（SSD）上划出一块区域伪装成内存（称为虚拟内存/Swap）。

- **代价：** 电脑会瞬间变得像 PPT 一样卡顿，因为硬盘太慢了。但它**不会死机**，程序还能苟活。
    

#### 2. GPU 显存（VRAM）的绝路：硬件设计决定了无法妥协

GPU 的工作方式和 CPU 完全不同。CPU 像一个智商极高的老教授，一次只解一道微积分（单核串行）；而 GPU 内部是由成千上万个“小学生”组成的庞大矩阵（多核并行），它们同时开工，在极短时间内完成几万个矩阵乘法。

这种恐怖的并发计算对数据传输速度（带宽）提出了变态的要求：

- **英伟达 H100 的显存（HBM3）带宽**：高达 **3.35 TB/s**。
    
- **主板上的 PCIe 5.0 总线速度**：只有 **128 GB/s**。
    

如果大模型在运行时显存满了，GPU 绝对无法像 CPU 那样把数据丢到内存或硬盘里去“边读边算”——因为 PCIe 总线的传输速度比显存慢了数十倍。这就好比让一个时速 300 公里的高铁，去等一个骑共享单车送零件的快递员，整个计算流水线会瞬间发生严重的阻塞。

因此，英伟达的 CUDA 驱动在设计时非常决绝：**一旦申请的显存空间超过了物理显存的极限，直接抛出 `RuntimeError: CUDA out of memory` 并强行终止程序。** 宁可直接崩溃，也绝不妥协去用慢速设备。
