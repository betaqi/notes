## 一、前置认知：下载不是 `wget`，是“缓存系统”

新手最大的误解是把模型下载当成“下个大文件”。实际上 Hugging Face / ModelScope 工具链背后是一整套**带缓存、带去重、带版本、带校验**的系统。理解这套系统，才能解释一堆“灵异现象”：

- 为什么下完模型，磁盘里却找不到 `config.json`？（答案：在 cache 里，`local-dir` 是软链接）
- 为什么换个脚本又重新下了一遍？（答案：缓存目录变了 / `local_dir` 用法不对）
- 为什么 `git clone` 下来全是几 KB 的小文件？（答案：没装 git-lfs）
- 为什么团队五个人每人一份 70GB 的重复模型？（答案：没规划共享 `HF_HOME`）

| 你以为的下载 | 实际的下载 |
| --- | --- |
| 一个 URL → 一个文件 | 一个 repo → 多文件 + index + 校验 |
| 下完就完事 | 落到 cache，按 commit hash 归档 |
| 重下就重新下 | 命中缓存秒返回，断点续传 |
| 文件就是文件 | blob 内容寻址 + snapshot 软链接 |

> **核心心智模型**：下载工具 = 包管理器（npm/pip）。它管的是“缓存 + 版本 + 链接”，不是“一个文件”。

## 二、三种下载方式全解

### 2.1 方式一：`huggingface-cli download`（推荐主力）

官方 CLI，**断点续传 + 多线程 + 自动处理 LFS + 缓存去重**，工程首选。

```bash
# 安装
pip install -U "huggingface_hub[cli]"

# 最简：下整个 repo 到默认缓存
huggingface-cli download Qwen/Qwen2.5-7B-Instruct

# 下到指定目录（注意 local-dir 的行为，见第五章）
huggingface-cli download Qwen/Qwen2.5-7B-Instruct --local-dir ./Qwen2.5-7B

# 只下需要的文件（省流量，跳过无关的 .bin / 图片）
huggingface-cli download Qwen/Qwen2.5-7B-Instruct \
  --include "*.safetensors" "*.json" "tokenizer*" \
  --exclude "*.pth" "*.bin"

# 锁定版本（生产必须）
huggingface-cli download Qwen/Qwen2.5-7B-Instruct --revision <commit-hash>

# 下数据集要加 --repo-type
huggingface-cli download tatsu-lab/alpaca --repo-type dataset
```

**常用参数速查：**

| 参数 | 作用 |
| --- | --- |
| `--local-dir DIR` | 落地到指定目录（默认走缓存） |
| `--include / --exclude` | 通配符筛选文件，省流量关键 |
| `--revision` | branch / tag / commit hash |
| `--repo-type` | `model`(默认) / `dataset` / `space` |
| `--cache-dir` | 临时改缓存位置 |
| `--token` | 私有/门控模型的访问令牌 |
| `--max-workers` | 并发下载线程数 |

### 2.2 方式二：`modelscope download`（国内首选）

阿里 ModelScope，国内带宽友好，用法几乎和 HF CLI 一一对应。

```bash
pip install -U modelscope

# 下整个模型
modelscope download --model Qwen/Qwen2.5-7B-Instruct

# 指定落地目录
modelscope download --model Qwen/Qwen2.5-7B-Instruct --local_dir ./Qwen2.5-7B

# 只下部分文件
modelscope download --model Qwen/Qwen2.5-7B-Instruct config.json model.safetensors.index.json
```

> 注意：HF 用 `--local-dir`（连字符），ModelScope 用 `--local_dir`（下划线）。两边参数风格不统一，是高频笔误点。

### 2.3 方式三：`git lfs clone`（不推荐，但要会）

HF/ModelScope 仓库底层就是 Git + Git LFS，可以直接 clone。**但这是最差的方式**：慢、占双倍空间（`.git` 里一份 + 工作区一份）、没断点续传。

```bash
# 必须先装 git-lfs，否则下下来全是指针文件
git lfs install

git clone https://huggingface.co/Qwen/Qwen2.5-7B-Instruct

# 进阶：先克隆指针，再按需拉大文件（省空间）
GIT_LFS_SKIP_SMUDGE=1 git clone https://huggingface.co/Qwen/Qwen2.5-7B-Instruct
cd Qwen2.5-7B-Instruct
git lfs pull --include "*.safetensors"
```

> **经典坑**：没 `git lfs install` 就 clone，权重文件全是几 KB 的文本指针（内容是 `version https://git-lfs...`），加载时报“文件损坏 / 不是合法 safetensors”。

### 2.4 三者对比与选型

| 维度 | huggingface-cli | modelscope | git lfs clone |
| --- | --- | --- | --- |
| 断点续传 | ✅ | ✅ | ❌ |
| 多线程 | ✅ | ✅ | ❌ |
| 文件筛选 | ✅ include/exclude | ✅ | ❌（要 skip-smudge 绕） |
| 占用空间 | 1 份 | 1 份 | **2 份**（含 .git） |
| 缓存去重 | ✅ | ✅ | ❌ |
| 国内速度 | 需配镜像 | 原生快 | 慢 |
| **推荐度** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐(国内) | ⭐ |

**口诀：国内用 modelscope，国外/通用用 huggingface-cli，git clone 只在万不得已时用。**

## 三、缓存路径约定（搞懂硬盘上到底放了什么）

### 3.1 HF 缓存的环境变量层级

Hugging Face 缓存路径由一组环境变量逐层决定，**优先级从外到内**：

| 环境变量 | 作用 | 默认值 |
| --- | --- | --- |
| `HF_HOME` | HF 一切数据的根目录（最常改的） | `~/.cache/huggingface` |
| `HF_HUB_CACHE` | 模型/数据集缓存目录 | `$HF_HOME/hub` |
| `HF_ASSETS_CACHE` | 库生成的中间产物 | `$HF_HOME/assets` |
| `HF_TOKEN` | 访问令牌 | 无 |

```bash
# 最常见的一行配置：把所有 HF 数据集中到大硬盘
export HF_HOME=/data/huggingface

# 等价于让 hub 缓存落在 /data/huggingface/hub
```

> **实战建议**：只记住 `HF_HOME` 一个就够 90% 场景。把它指到大容量盘（模型动辄几十 GB，别堆在系统盘 `~`）。

### 3.2 缓存目录的内部结构（内容寻址）

这是最该看懂的一张图。下载 `Qwen/Qwen2.5-7B-Instruct` 后，`$HF_HUB_CACHE` 长这样：

```
hub/
└── models--Qwen--Qwen2.5-7B-Instruct/   # 组织名/模型名 中的 / 换成 --
    ├── blobs/                            # 真实文件，按内容哈希命名
    │   ├── a1b2c3...                     # 一个 safetensors 分片
    │   └── d4e5f6...                     # config.json 内容
    ├── snapshots/                        # 每个 commit 一个目录
    │   └── <commit-hash>/                # 里面全是软链接 → blobs/
    │       ├── config.json -> ../../blobs/d4e5f6...
    │       └── model-00001-of-00004.safetensors -> ../../blobs/a1b2c3...
    └── refs/
        └── main                          # 记录 main 分支当前指向哪个 commit
```

**三层机制：**
1. `blobs/`：真正的二进制，**按内容哈希命名**（内容寻址）。同样内容只存一份。
2. `snapshots/<commit>/`：你“看到”的目录结构，文件全是指向 blobs 的**软链接**。
3. `refs/`：分支名 → commit 的映射，等价于 git 的 refs。

> **去重的魔法**：两个模型若共享同一个文件（如相同的 tokenizer），blobs 里只存一份。同一模型更新版本，没变的分片也不会重复下载。这就是为什么缓存系统比裸下载省空间。

### 3.3 ModelScope 缓存

| 环境变量 | 作用 | 默认值 |
| --- | --- | --- |
| `MODELSCOPE_CACHE` | ModelScope 缓存根目录 | `~/.cache/modelscope/hub` |

```bash
export MODELSCOPE_CACHE=/data/modelscope
```

结构与 HF 类似（也有 blobs/snapshots 思路），但目录命名是 `模型路径` 直观铺开，没 HF 那么“反直觉”。

## 四、国内镜像与代理配置

### 4.1 `HF_ENDPOINT` —— 国内开发铁律

直连 `huggingface.co` 在国内经常 403/超时。把端点换成镜像站 `hf-mirror.com` 即可复用整套 HF 工具链：

```bash
# 临时（当前终端）
export HF_ENDPOINT=https://hf-mirror.com
huggingface-cli download Qwen/Qwen2.5-7B-Instruct

# 永久：写进 ~/.bashrc 或 ~/.zshrc
echo 'export HF_ENDPOINT=https://hf-mirror.com' >> ~/.bashrc
source ~/.bashrc
```

> **关键认知**：`HF_ENDPOINT` 不改你的代码，只改下载工具去哪个域名拉数据。镜像站只是 HF 的只读副本，repo 名、文件结构完全一致。

### 4.2 代理配置（有梯子时）

```bash
export https_proxy=http://127.0.0.1:7890
export http_proxy=http://127.0.0.1:7890
```

> **二选一原则**：用了 `HF_ENDPOINT` 镜像通常就不需要代理；用代理直连官方就别再设镜像。两个一起配反而可能冲突。

### 4.3 门控模型（Gated）与 Token

Llama、Gemma 等需要在官网“同意协议”后才能下，下载时要带 token：

```bash
huggingface-cli login              # 交互式输入 token，存到本地
# 或
export HF_TOKEN=hf_xxxxxxxx
huggingface-cli download meta-llama/Llama-3.1-8B-Instruct
```

> Token 在 HF 网站 Settings → Access Tokens 生成。镜像站一般不需要 token，但门控模型镜像可能不全。

## 五、`snapshot_download` 与 `local_dir`：软链接 vs 实文件（最易混点）

### 5.1 两种落地模式

`huggingface-cli download` 和 Python 的 `snapshot_download` 本质是同一套逻辑，关键看**有没有指定 `local_dir`**：

```python
from huggingface_hub import snapshot_download

# 模式 A：不指定 local_dir → 落到缓存，返回 snapshot 目录路径
path = snapshot_download("Qwen/Qwen2.5-7B-Instruct")
# path = ~/.cache/huggingface/hub/models--Qwen--.../snapshots/<commit>/
# 目录里全是软链接 → blobs/

# 模式 B：指定 local_dir → 落到你的目录
path = snapshot_download("Qwen/Qwen2.5-7B-Instruct", local_dir="./Qwen2.5-7B")
# path = ./Qwen2.5-7B/，里面是真实文件（新版默认实拷贝）
```

### 5.2 两种模式的差异表

| 维度 | 缓存模式（无 local_dir） | local_dir 模式 |
| --- | --- | --- |
| 落地位置 | `$HF_HOME/hub/...` | 你指定的目录 |
| 文件形态 | 软链接 → blobs | 真实文件（新版）/ 软链接（旧版） |
| 多模型去重 | ✅ 自动 | ❌ 各占一份 |
| 适合场景 | 日常开发、多模型 | 打包、拷贝、上传服务器、Docker |
| 删除影响 | 删 snapshot 不删 blob | 直接删真文件 |

### 5.3 历史坑：旧版 `local_dir` 的软链接

旧版本 `huggingface_hub` 即便指定了 `local_dir`，目录里也是**软链接指回缓存**——你以为复制走了，结果 tar 打包过去全是死链接。

- 旧版可用 `--local-dir-use-symlinks False` 强制实拷贝。
- 新版（`huggingface_hub>=0.23` 起）`local_dir` 默认就是实文件，该参数已废弃。

> **结论**：要“一个干净、可搬运、自包含的模型目录”（拷贝到离线机器 / 打进 Docker 镜像 / 上传 OSS），就用 `local_dir`，并确认版本够新或加 `--local-dir-use-symlinks False`。日常本机开发就用默认缓存模式，省空间。

## 六、断点续传与 Hash 校验

### 6.1 断点续传

`huggingface-cli` 和 `modelscope` 默认支持断点续传——下到一半断网，重跑同一条命令会接着下，不会从头来。底层靠 `.incomplete` 临时文件记录进度。

```bash
# 断了直接重跑，自动续传
huggingface-cli download Qwen/Qwen2.5-7B-Instruct --local-dir ./Qwen2.5-7B
```

> 这正是不推荐 `git clone` 的原因之一：git clone 中断后往往要整个重来。

### 6.2 Hash 校验（完整性保证）

HF 在传输层用 **ETag / SHA256** 校验每个文件，下载完会自动比对，损坏的分片会被丢弃重下。你通常无需手动校验，但生产环境想主动核验时：

```bash
# 查看某文件的官方哈希（页面或 API），与本地比对
sha256sum model-00001-of-00004.safetensors
```

更稳妥的是**锁 commit hash 下载**——只要 commit 一致，内容字节级一致：

```bash
huggingface-cli download Qwen/Qwen2.5-7B-Instruct --revision <commit-hash> --local-dir ./m
```

## 七、模型完整性自检：齐了才能推理

下完别急着跑，先确认“推理必需文件”是否齐全。**缺一个都加载失败**。

### 7.1 safetensors 模型（Transformers / vLLM）必需清单

| 文件 | 作用 | 缺了会怎样 |
| --- | --- | --- |
| `config.json` | 模型架构 | 无法实例化模型类 |
| `model*.safetensors` | 权重（可能多分片） | 没权重，直接报错 |
| `model.safetensors.index.json` | 分片索引（**仅多分片时**） | 多分片缺它无法拼装 |
| `tokenizer.json` 或 `tokenizer.model` | 分词器 | 无法编码输入 |
| `tokenizer_config.json` | 分词器配置 + chat template | 对话格式错乱 |
| `generation_config.json` | 生成默认参数（可选） | 用内置默认值，不致命 |
| `special_tokens_map.json` | 特殊 token | 部分模型缺它会出错 |

> **分片自检**：`index.json` 里 `weight_map` 引用的每个 `.safetensors` 都必须在本地存在。`-of-00004` 说明应有 4 片，数数对不对。

### 7.2 GGUF 模型（llama.cpp / Ollama）

GGUF 是**单文件自包含**（权重 + tokenizer + 元数据全在里面），自检最简单：

```bash
ls -lh *.gguf   # 有这一个文件、大小正常即可
```

### 7.3 一个快速自检脚本

```bash
#!/bin/bash
# 用法: ./check.sh ./Qwen2.5-7B
DIR=$1
for f in config.json tokenizer_config.json; do
  [ -f "$DIR/$f" ] && echo "OK  $f" || echo "MISSING  $f"
done
# 检查是否有权重
ls "$DIR"/*.safetensors >/dev/null 2>&1 && echo "OK  weights" || echo "MISSING  weights"
# 多分片则校验 index
if ls "$DIR"/*-of-*.safetensors >/dev/null 2>&1; then
  [ -f "$DIR/model.safetensors.index.json" ] && echo "OK  index" || echo "MISSING  index"
fi
```

## 八、多模型共享缓存的目录规划（团队/服务器实战）

### 8.1 问题：人均一份 70GB 的浪费

默认每个用户的模型都下到自己的 `~/.cache`，团队 5 人就是 5 份重复。解决思路是**共享缓存 + 内容寻址去重**。

### 8.2 方案：集中式共享 HF_HOME

```bash
# 在服务器上建一个共享缓存目录，所有用户指向它
sudo mkdir -p /data/shared/hf
sudo chmod -R 2775 /data/shared/hf      # setgid，新文件继承组
sudo chgrp -R ai-team /data/shared/hf   # 团队组可读写

# 每个用户 / 每个服务的环境变量都指过去
export HF_HOME=/data/shared/hf
```

**收益：**
- 同一模型只下一次，全员共用 snapshots/blobs。
- 不同模型间相同文件（tokenizer 等）自动去重。
- 新成员零下载，直接命中缓存。

### 8.3 权限与并发注意

| 风险 | 对策 |
| --- | --- |
| 多人同时下同一模型 | HF 有文件锁，安全；但建议错峰 |
| 权限不一致导致无法写 | 用 setgid + 统一组 |
| 有人误删 blobs | 共享缓存设只读 + 专人维护下载 |

> **进阶**：生产可搭**私有 mirror**（如 `hf-mirror` 自建、或对象存储 + nginx），所有机器从内网拉，既快又稳，且锁版本不怕上游漂移。

## 九、离线环境的模型分发策略

很多企业的训练/推理机房**完全不通外网**（安全合规要求）。模型必须从联网机器“摆渡”进去。

### 9.1 标准三步走

```
联网机器 (下载/打包)  →  传输介质/内网  →  离线机器 (解包/加载)
```

**第一步：联网机下载成自包含目录（务必用 local_dir 实文件）**

```bash
huggingface-cli download Qwen/Qwen2.5-7B-Instruct \
  --local-dir ./Qwen2.5-7B \
  --local-dir-use-symlinks False    # 旧版需要，确保是实文件不是软链接
tar -czf Qwen2.5-7B.tar.gz ./Qwen2.5-7B
```

**第二步：传输**（scp / U 盘 / 内网对象存储 / 私有制品库）

```bash
scp Qwen2.5-7B.tar.gz user@offline-server:/data/models/
```

**第三步：离线机解包 + 强制离线模式加载**

```bash
tar -xzf Qwen2.5-7B.tar.gz -C /data/models/
export HF_HUB_OFFLINE=1     # 禁止任何联网检查，纯本地加载
export TRANSFORMERS_OFFLINE=1
```

### 9.2 离线关键开关

| 环境变量 | 作用 |
| --- | --- |
| `HF_HUB_OFFLINE=1` | hub 完全离线，不发任何网络请求 |
| `TRANSFORMERS_OFFLINE=1` | Transformers 库离线模式 |

> **离线第一坑**：忘了设 `HF_HUB_OFFLINE=1`，库仍会去 HF 检查更新 → 超时卡住几十秒甚至报错。离线机务必把这俩写进环境。

### 9.3 加载时用绝对路径，别用 repo id

离线加载一律传**本地目录路径**，而非 `Qwen/Qwen2.5-7B`（那会触发联网解析）：

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
model = AutoModelForCausalLM.from_pretrained("/data/models/Qwen2.5-7B")  # ✅ 本地路径
tok = AutoTokenizer.from_pretrained("/data/models/Qwen2.5-7B")
```

### 9.4 也可摆渡整个缓存目录

若离线机用 repo id 加载，可把联网机的 `$HF_HOME/hub/models--Qwen--Qwen2.5-7B-Instruct` 整个目录（含 blobs/snapshots/refs）打包过去，放到离线机相同 `HF_HOME` 下，配合 `HF_HUB_OFFLINE=1` 即可用 repo id 命中缓存。

## 十一、本节核心认知（必须内化）

| # | 认知 |
| --- | --- |
| 1 | 下载工具是**缓存系统**，不是 wget：blobs(实文件) + snapshots(软链接) + refs |
| 2 | 主力 `huggingface-cli download`，国内 `modelscope download`，`git clone` 尽量别用 |
| 3 | `HF_HOME` 指到大盘是第一件事；只记这一个变量够用 |
| 4 | 国内必配 `HF_ENDPOINT=https://hf-mirror.com` |
| 5 | 要**可搬运的干净目录**用 `local_dir`（实文件）；日常省空间用默认缓存 |
| 6 | 断点续传默认开；锁 `--revision <commit>` 保字节级一致 |
| 7 | 推理前自检：config + 权重 + index(多分片) + tokenizer 缺一不可 |
| 8 | 团队共享一个 `HF_HOME` 去重，省下成倍磁盘 |
| 9 | 离线机：`local_dir` 打包 + `HF_HUB_OFFLINE=1` + 本地路径加载 |
| 10 | git clone 前必 `git lfs install`，否则全是指针文件 |
