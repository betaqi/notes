## 一、前置认知：为什么要先学“模型仓库生态”？

在写第一行推理代码之前，你必须先回答三个工程问题：

| 问题        | 涉及知识            |
| --------- | --------------- |
| 模型从哪来？    | 分发渠道、镜像、下载工具    |
| 下下来是什么？   | 标准文件、权重格式、分片    |
| 能不能用、怎么用？ | License、量化、引擎选型 |

整个 AI 工程的“供应链起点”就是模型仓库。下游的 vLLM、Ollama、LangChain、RAG、Agent 全部建立在这个起点上。**搞不清模型文件，后面所有部署/推理代码都是抄作业**。

> 类比：模型仓库之于 AI 工程，相当于 npm/PyPI 之于普通软件工程。但因为模型动辄几十 GB，又有量化、分片、license 等复杂度，远比普通包管理麻烦。

## 二、四大模型分发渠道

### 2.1 Hugging Face Hub —— 全球事实标准

地址：`https://huggingface.co`

Hugging Face 是 AI 界的 GitHub，三大资源类型：

| 资源类型        | 含义                        | 例子                             | URL 形式                                   |
| ----------- | ------------------------- | ------------------------------ | ---------------------------------------- |
| **Model**   | 模型权重 + 配置 + tokenizer     | `Qwen/Qwen2.5-7B-Instruct`     | huggingface.co/Qwen/Qwen2.5-7B-Instruct  |
| **Dataset** | 训练/评测数据集                  | `tatsu-lab/alpaca`             | huggingface.co/datasets/tatsu-lab/alpaca |
| **Space**   | 在线 Demo（Gradio/Streamlit） | `stabilityai/stable-diffusion` | huggingface.co/spaces/...                |

仓库命名规范：`组织名/模型名`（例 `meta-llama/Llama-3.1-8B-Instruct`）。

**三种下载方式：**

```bash
# 方式 1：CLI（推荐，断点续传 + 多线程）
huggingface-cli download Qwen/Qwen2.5-7B-Instruct --local-dir ./Qwen2.5-7B

# 方式 2：Python SDK（脚本里用）
from huggingface_hub import snapshot_download
snapshot_download("Qwen/Qwen2.5-7B-Instruct", local_dir="./Qwen2.5-7B")

# 方式 3：git lfs（不推荐，慢且占双倍空间）
git lfs install
git clone https://huggingface.co/Qwen/Qwen2.5-7B-Instruct
```

**Model Card（README.md）必读章节：**
- *Model Description*：参数量、训练数据、用途
- *Usage*：示例代码（决定你抄哪段开始）
- *Evaluation*：基准测试得分
- *License*：商用边界（最重要）

### 2.2 ModelScope —— 国内首选镜像

地址：`https://modelscope.cn`

阿里达摩院出品，国内带宽友好，主流开源模型几乎都有同步副本。**国内开发者首选**。

```bash
pip install modelscope
modelscope download --model Qwen/Qwen2.5-7B-Instruct --local_dir ./Qwen2.5-7B
```

或者用 HF 镜像变量直接复用 Hugging Face 工具链（更通用）：

```bash
export HF_ENDPOINT=https://hf-mirror.com
huggingface-cli download Qwen/Qwen2.5-7B-Instruct
```

> **国内开发铁律**：`HF_ENDPOINT` 这个环境变量必须配上，否则下载经常 403/超时。生产服务器建议写到 `~/.bashrc`。

### 2.3 Ollama Library —— 开箱即用

地址：`https://ollama.com/library`

Ollama 自带的模型仓库，**只发布 GGUF 量化版**，零配置一键运行：

```bash
ollama pull qwen2.5:7b      # 默认 Q4_K_M
ollama pull qwen2.5:7b-q8_0 # 指定量化等级
ollama run qwen2.5:7b
```

适合：本地试用、笔记本部署、教学 demo、不想折腾 CUDA 的场景。**生产环境一般不用 Ollama**，因为吞吐有限。

### 2.4 其他渠道

| 渠道 | 特点 | 适用场景 |
| --- | --- | --- |
| **Kaggle Models** | Google 系生态，Gemma、TPU 版本常驻 | 想跑 Gemma / TPU |
| **Replicate** | 在线 API 调用，按调用次数付费 | 临时 demo，不想本地跑 |
| **LM Studio Hub** | LM Studio 内置仓库（GGUF/MLX） | mac 桌面用户 |
| **厂商官方** | Meta、Mistral、智谱、月之暗面官网 | 第一时间拿到原版 |

**优先级建议：**
- 国内 → ModelScope > HF 镜像 > Ollama
- 国外 → HF Hub > Ollama
- mac 本地 → MLX 社区 / Ollama
- 生产部署 → 一定锁定 commit hash，自建私有 mirror

## 四、五大标准配置文件详解

### 4.1 `config.json` — 模型的“身份证”

定义模型架构、层数、隐藏维度、词表大小等。Transformers 通过它决定加载哪个模型类。

```json
{
  "architectures": ["Qwen2ForCausalLM"],
  "model_type": "qwen2",
  "hidden_size": 3584,
  "intermediate_size": 18944,
  "num_hidden_layers": 28,
  "num_attention_heads": 28,
  "num_key_value_heads": 4,
  "vocab_size": 152064,
  "max_position_embeddings": 32768,
  "rope_theta": 1000000.0,
  "torch_dtype": "bfloat16",
  "tie_word_embeddings": false
}
```

**看 config.json 你能立刻判断：**

| 字段 | 能读出什么 |
| --- | --- |
| `architectures` | 模型类（决定能不能跑） |
| `model_type` | 框架族（qwen2 / llama / mistral …） |
| `hidden_size × num_hidden_layers` | 大致参数规模 |
| `num_key_value_heads < num_attention_heads` | 用了 GQA（分组查询注意力，省 KV Cache） |
| `max_position_embeddings` | 原生上下文长度（这里是 32K） |
| `rope_theta` | RoPE 基底，决定长文本外推能力 |
| `torch_dtype` | 原生精度（bf16 / fp16 / fp32） |
| `tie_word_embeddings` | 是否共享 embedding 与输出层权重 |

### 4.2 `tokenizer.json` + `tokenizer_config.json`

- **`tokenizer.json`**：HuggingFace fast tokenizer 序列化文件，**单文件即可加载**，Rust 实现的 tokenizers 库会读它。
- **`tokenizer_config.json`**：tokenizer 元信息

```json
{
  "bos_token": null,
  "eos_token": "<|im_end|>",
  "pad_token": "<|endoftext|>",
  "model_max_length": 131072,
  "tokenizer_class": "Qwen2Tokenizer",
  "chat_template": "..."
}
```

> **关键 tokens**：
> - `bos_token`：Begin of Sequence
> - `eos_token`：End of Sequence（生成停止信号）
> - `pad_token`：批量推理时的填充
> - `unk_token`：未知 token

### 4.3 `chat_template.jinja` — 对话格式的“翻译官”

把 `messages` 数组渲染成模型实际看到的文本字符串。**这是初学者最容易踩坑的点**：同一段 messages，不同模型用不同模板，渲染结果完全不同。

例如 Qwen2 的模板：

```jinja
{%- for message in messages %}
<|im_start|>{{ message.role }}
{{ message.content }}<|im_end|>
{%- endfor %}
{%- if add_generation_prompt %}
<|im_start|>assistant
{%- endif %}
```

而 Llama 3 是这样：

```
<|begin_of_text|><|start_header_id|>user<|end_header_id|>

Hello<|eot_id|><|start_header_id|>assistant<|end_header_id|>

```

**核心认知**：你调 `chat/completions` 时，框架内部干的事 = `messages` + `chat_template` → 渲染成纯文本 → tokenize → 喂给 generate。

> 部分新版 Transformers 会把模板单独抽到 `chat_template.jinja`，部分仍嵌在 `tokenizer_config.json["chat_template"]`，两种都要会读。

### 4.4 `generation_config.json` — 生成默认参数

```json
{
  "do_sample": true,
  "temperature": 0.7,
  "top_p": 0.8,
  "top_k": 20,
  "repetition_penalty": 1.05,
  "max_new_tokens": 2048,
  "eos_token_id": [151645, 151643],
  "pad_token_id": 151643
}
```

调用时若没传 `temperature`、`top_p` 等，就用这里的默认值。**注意 `eos_token_id` 是数组**，因为 Qwen 有多个停止符。

### 4.5 `special_tokens_map.json`

```json
{
  "additional_special_tokens": ["<|im_start|>", "<|im_end|>", "<|tool_call|>"],
  "eos_token": "<|im_end|>",
  "pad_token": "<|endoftext|>"
}
```

记录所有“特殊 token”——这些 token 不参与正常分词，整体作为一个单元处理（如对话边界、工具调用标记）。

## 五、权重格式全解（最高频判别点）

同一个模型可能存在多种格式，每种对应不同推理引擎。**格式决定引擎，引擎决定可行性**。

| 格式 | 文件 | 主流引擎 | 量化 | 特点 |
| --- | --- | --- | --- | --- |
| **safetensors** | `.safetensors` | Transformers / vLLM / SGLang | 否 | 当前主流，零拷贝、无代码执行风险 |
| **PyTorch bin** | `pytorch_model.bin` | Transformers | 否 | 旧格式，pickle，有反序列化风险 |
| **GGUF** | `.gguf` | llama.cpp / Ollama / LM Studio | 是 | CPU/Metal 友好，单文件含权重+tokenizer |
| **MLX** | `.safetensors`(MLX 命名空间) | mlx-lm | 可量化 | Apple Silicon 专用 |
| **AWQ** | `.safetensors` + `quant_config.json` | vLLM / AutoAWQ | 4-bit | 激活感知量化，精度损失小 |
| **GPTQ** | `.safetensors` + `quantize_config.json` | vLLM / AutoGPTQ | 4-bit | 训练后量化，速度快 |

**判别口诀：**

```
看到 .gguf                      → llama.cpp / Ollama
看到 .safetensors + 仓名带 AWQ  → vLLM 量化版
看到 .safetensors + 仓名带 GPTQ → vLLM 量化版
看到 .safetensors（无量化标记） → Transformers / vLLM 原生精度
看到 mlx-community/xxx          → 苹果 M 系列芯片专用
看到 pytorch_model.bin          → 老仓库（2023 之前），尽量找 safetensors 版本
```

### 5.1 为什么 safetensors 取代 .bin

| 维度 | .bin (pickle) | safetensors |
| --- | --- | --- |
| 加载速度 | 慢，需反序列化 | 快，零拷贝 mmap |
| 安全性 | **可执行任意代码** | 仅纯数据 |
| 跨语言 | 只支持 Python | Rust/Python/JS 都支持 |
| 元数据 | 散落各处 | 内嵌 header |

> **结论**：新模型一律 safetensors。遇到 `.bin` 几乎都是 2023 年前的老仓库，要警惕反序列化攻击。

### 5.2 量化格式与引擎对应

| 量化 | 位宽 | 显存收益 | 推荐引擎 | 注意 |
| --- | --- | --- | --- | --- |
| FP16 / BF16 | 16-bit | 基线 | Transformers/vLLM | 原生精度 |
| INT8 | 8-bit | ~50% ↓ | vLLM / Transformers | 几乎无损 |
| AWQ (W4A16) | 4-bit 权重 | ~75% ↓ | vLLM | 业界优选 |
| GPTQ (W4A16) | 4-bit 权重 | ~75% ↓ | vLLM | 略逊于 AWQ |
| GGUF Q4_K_M | 混合 4-bit | ~75% ↓ | llama.cpp/Ollama | CPU 推理首选 |

## 六、分片机制（Shard）

大模型权重一个文件装不下（HF 单文件上限约 50GB，实际通常切 2~10GB 一片）：

```
model-00001-of-00004.safetensors    # 第 1 片
model-00002-of-00004.safetensors    # 第 2 片
model-00003-of-00004.safetensors    # 第 3 片
model-00004-of-00004.safetensors    # 第 4 片
model.safetensors.index.json        # 索引
```

`model.safetensors.index.json` 大致结构：

```json
{
  "metadata": { "total_size": 15231233024 },
  "weight_map": {
    "model.embed_tokens.weight": "model-00001-of-00004.safetensors",
    "model.layers.0.self_attn.q_proj.weight": "model-00001-of-00004.safetensors",
    "model.layers.27.mlp.down_proj.weight": "model-00004-of-00004.safetensors",
    "lm_head.weight": "model-00004-of-00004.safetensors"
  }
}
```

工作机制：
1. 加载器先读 `index.json`
2. 根据 `weight_map` 逐个权重去对应分片中读取
3. 分片可以并行加载，加速冷启动

> **踩坑提醒**：下载时**必须把所有分片+index.json 都下完**，少一片直接加载失败。`huggingface-cli download` 会自动一起下，手动 wget 才会缺。

## 七、版本管理：revision / branch / tag + LFS

Hugging Face 仓库底层就是 **Git + Git LFS**，所以拥有完整的版本能力：

```python
from huggingface_hub import snapshot_download

# revision 可以是 branch / tag / commit hash
snapshot_download("Qwen/Qwen2.5-7B-Instruct", revision="main")
snapshot_download("Qwen/Qwen2.5-7B-Instruct", revision="v1.0")
snapshot_download("Qwen/Qwen2.5-7B-Instruct", revision="abc123def456")
```

**生产建议**：始终锁定具体 commit hash，避免上游悄悄更新权重导致行为漂移。这等价于 npm 锁版本。

### 关于 LFS（Large File Storage）

- 大于 10MB 的文件走 LFS（git 仓库本体只存指针）
- `git clone` 时如果**没装 git-lfs**，下下来的是几 KB 的指针文件而不是真权重
- 这是新手最经典的坑：以为下完了，加载时报“文件损坏”

```bash
# 确保安装
git lfs install --skip-smudge  # 先克隆指针
cd Qwen2.5-7B-Instruct
git lfs pull                   # 再拉真实大文件
```

但更推荐的做法是**直接用 `huggingface-cli download`**，原生处理 LFS，且支持断点续传与多线程。

## 八、许可证（License）—— 商用边界

**重要程度：⭐⭐⭐⭐⭐**。在企业里用错许可证可能直接吃官司。

| License | 商用 | 要求 | 代表模型 |
| --- | --- | --- | --- |
| **Apache 2.0** | ✅ 完全开放 | 保留版权声明 | Qwen2.5（≤72B 多数）、Mistral 7B、DeepSeek-V2 |

| **Qwen License** | ⚠️ 有条件 | Qwen 72B 商用需申请 | Qwen 72B / Qwen-Max |
| **Gemma Terms of Use** | ⚠️ 有条件 | 禁止特定用途清单（武器/虐待/欺诈） | Gemma 2 / 3 |

**判别动作（三处对齐才能下结论）：**

3. 模型卡“License”章节

## 九、GGUF 量化等级速查

GGUF 是 Ollama / llama.cpp 生态的格式，量化级别繁多。**记住下面这张表就够用 90% 场景**。

以 7B 模型为例（不同模型略有差异）：

| 量化等级 | 位宽 | 显存/内存（7B） | 质量损失 | 推荐场景 |
| --- | --- | --- | --- | --- |
| **Q2_K** | 2-bit | ~3.0 GB | 严重 | 极限低显存、可玩性差 |
| **Q3_K_M** | 3-bit | ~3.6 GB | 明显 | 不推荐 |
| **Q4_0** | 4-bit | ~4.1 GB | 一般 | 老式量化，已淘汰 |
| **Q4_K_M** | 4-bit | ~4.4 GB | 轻微 | ⭐ **默认首选** |
| **Q5_K_M** | 5-bit | ~5.1 GB | 几乎无 | 质量优先、显存够 |
| **Q6_K** | 6-bit | ~5.9 GB | 几乎无 | 接近 fp16 |
| **Q8_0** | 8-bit | ~7.7 GB | 无 | 高保真、做评测 |
| **F16** | 16-bit | ~14 GB | 无 | 原始精度 |

**口诀：**
- 默认 **Q4_K_M**（甜点平衡）
- 显存够就升 **Q5_K_M / Q6_K**
- 做评测对比 → **Q8_0 / F16**
- 极限内存（手机/树莓派）→ Q3_K_M / Q2_K（牺牲质量）

**命名解读：** `Q{位宽}_{方案}_{尺寸}`
- `K` = K-quants（按 block 量化，比老式 Q4_0 质量好）
- `S/M/L` = Small/Medium/Large（精度从低到高）

> 经验数据：Q4_K_M 相对 fp16 在 MMLU、HumanEval 等基准上通常掉 1~3 个百分点，几乎察觉不到。

## 十、综合实战：30 秒读懂任意仓库

打开 `Qwen/Qwen2.5-7B-Instruct-AWQ` 仓库，看到下列文件：

```
config.json
generation_config.json
tokenizer.json
tokenizer_config.json
model-00001-of-00002.safetensors
model-00002-of-00002.safetensors
model.safetensors.index.json
quant_config.json          ← AWQ 标志
LICENSE  (Apache 2.0)
```

**30 秒推断步骤：**

1. 仓名带 `AWQ` + 存在 `quant_config.json` → **4-bit AWQ 量化版**
2. 文件是 `.safetensors` → 用 **vLLM / AutoAWQ** 跑，**不能**用 Ollama
3. 分 2 片 → 总大小约 5GB（7B 4-bit 的典型体积）
4. License = Apache 2.0 → **可商用**
5. 有 `tokenizer_config.json`（含 chat template）→ 可以直接走 `chat/completions`
6. 看 `config.json`：`max_position_embeddings=32768` → 原生 32K 上下文

→ 最终结论：**这是一个可商用的 Qwen2.5-7B AWQ 量化版，单卡 12GB 显存就能跑，用 vLLM 部署最佳。**

试试再分析下面这个仓库：

```
TheBloke/Llama-2-13B-chat-GGUF/
├── llama-2-13b-chat.Q4_K_M.gguf  (7.9 GB)
├── llama-2-13b-chat.Q5_K_M.gguf  (9.2 GB)
├── llama-2-13b-chat.Q8_0.gguf    (13.8 GB)
├── README.md
└── LICENSE  (Llama 2 Community License)
```

**推断：**
- `.gguf` → 用 Ollama / llama.cpp
- 13B + Q4_K_M ≈ 8GB → 16GB 显存机器够用
- License = Llama 2 Community → MAU 不超过 7 亿可商用，注意禁用清单
