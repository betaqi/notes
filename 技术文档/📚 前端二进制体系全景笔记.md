前端处理文件的核心流程通常是：`File (Blob)` $\rightarrow$ `ArrayBuffer` $\rightarrow$ `Uint8Array` $\rightarrow$ `操作/传输`。这是一个由**数据源**、**内存容器**、**操作视图**以及**转换桥梁**构成的分层体系。

## 一、 核心疑问：为什么需要 Uint8Array 包裹 ArrayBuffer？

在文件分片或读取场景中，我们经常看到这样的代码：

JavaScript

```
const chunkBuffer = fileData.slice(offset, offset + chunkSize); // 得到 ArrayBuffer
const chunkUint8 = new Uint8Array(chunkBuffer); // 🔥 必须包裹
```

**根本原因：**

- **ArrayBuffer 是“死”的（容器）**：它只是一块纯粹的固定长度的内存空间（可以想象为一块刚圈好的地皮）。它没有任何格式信息，JavaScript 无法通过下标直接读取或修改里面的内容。
    
- **Uint8Array 是“手”（视图 View）**：它是操作接口。它告诉 JS 引擎：“请把这块内存当成一个个 8 位无符号整数 (0-255) 来处理”。
    
- **算法依赖**：当你进行 MD5 计算（如 spark-md5）或数据校验时，算法需要遍历每一个字节。只有套上了 `Uint8Array` 这个视图，内存里的数据才是可遍历、可读写的。

## 二、 二进制家族全景图 (Cheat Sheet)

我们将这些概念分为三层：数据源层、内存层、视图层。

|**概念层级**|**形象比喻**|**核心特点 & 场景**|
|---|---|---|
|**Blob**|📦 密封快递箱|**数据源**。不可变、存在于磁盘或内存中。只能整体移动（上传）或切片，不能修改内部数据。<br><br>  <br><br>**场景**：`URL.createObjectURL` 图片预览、文件下载。|
|**File**|🏷️ 带标签的箱子|**数据源**。`Blob` 的子类。仅仅比 Blob 多了文件名 (`name`) 和最后修改时间 (`lastModified`)。<br><br>  <br><br>**场景**：`<input type="file">` 获取的对象。|
|**ArrayBuffer**|🧱 原始地皮|**内存层**。通用的二进制缓冲区。它是物理内存的映射，不能直接操作。<br><br>  <br><br>**场景**：核心存储容器，所有视图的“宿主”。|
|**Uint8Array**|👓 单色眼镜|**视图层**。最常用的视图。将内存看作字节 (Byte) 集合。<br><br>  <br><br>**场景**：文件切片处理、哈希计算、WebWorker 通信、Protobuf 序列化。|
|**DataView**|🔬 显微镜|**视图层**。高级视图。允许在同一段内存上读写不同类型（Int8, Float32等），并可控制字节序（大端/小端）。<br><br>  <br><br>**场景**：解析自定义二进制协议、解析图片/文件头结构。|
|**Base64**|📄 翻译文本|**转换层**。纯文本字符串。将二进制转为 64 个可打印字符，体积通常膨胀 33%。<br><br>  <br><br>**场景**：内嵌小图标 (Data URI)、JSON 中传输二进制数据。|

## 四、 字符串与二进制的转换桥梁

在计算机底层，所有东西都是数字（字节），而人类看到的是字符（String）。在 ArrayBuffer（二进制）和 String（字符串）之间，不能直接划等号（因为同一个汉字在 UTF-8 和 GBK 下对应的二进制完全不同）。

我们需要专门的“翻译官”：**TextDecoder** 和 **TextEncoder**。

### 1. 解码器：TextDecoder

- **作用**：将二进制数据 (`Uint8Array` / `ArrayBuffer`) 翻译成人类可读的字符串。
    
- **语法**：`new TextDecoder(label, options).decode(input)`
    
- **特点**：默认 `utf-8`，但也支持 `gbk` (解决老旧中文系统乱码的关键)。
    

JavaScript

```
// 假设这是 "你好" 的 UTF-8 二进制数据
const uint8Array = new Uint8Array([228, 189, 160, 229, 165, 189]);

// 1. 创建解码器
const decoder = new TextDecoder('utf-8');
// 2. 翻译
const str = decoder.decode(uint8Array);
console.log(str); // 输出: "你好"
```

> ⚠️ 警告：为什么不能用 String.fromCharCode？
> 
> 以前的老代码常用 String.fromCharCode(...uint8Array)，这是错误的做法！
> 
> fromCharCode 按 16 位读取，而现代 UTF-8 中文是 3 字节。直接把 UTF-8 字节扔进去会导致截断或错位，必须使用 TextDecoder。

### 2. 编码器：TextEncoder

- **作用**：将字符串翻译成二进制数据 (`Uint8Array`)。
    
- **语法**：`new TextEncoder().encode(input)`
    
- **注意**：只支持 UTF-8 编码。
    

JavaScript

```
const str = "Hello 世界";
const encoder = new TextEncoder();
const uint8Array = encoder.encode(str);
// 输出: Uint8Array(12) [72, 101, 108, 108, 111, 32, 228, 184, 150, 231, 149, 140]
```

### 3. 另一条路：Blob 与 FileReader

如果是文件对象（Blob/File），可以使用异步的 `FileReader`。

JavaScript

```
const blob = new Blob(['你好'], { type: 'text/plain' });
const reader = new FileReader();

// 告诉 FileReader 直接按文本读取，浏览器会自动处理编码
reader.readAsText(blob, 'utf-8'); 

reader.onload = () => {
  console.log(reader.result); // 输出: "你好"
};
```

### 4. 转换 API 对比表 (Cheat Sheet)

|**转换方向**|**API**|**特点**|**场景**|
|---|---|---|---|
|Binary $\rightarrow$ String|`new TextDecoder().decode()`|✅ **推荐**。同步，高性能，支持 UTF-8/GBK|解析 WebSocket 消息、解析文件内容片段|
|Binary $\rightarrow$ String|`FileReader.readAsText()`|异步，回调形式|直接读取用户上传的 .txt/.csv 文件|
|String $\rightarrow$ Binary|`new TextEncoder().encode()`|✅ **推荐**。同步，只生成 UTF-8|发送 WebSocket 消息、生成文件内容|
|Binary $\leftrightarrow$ String|`Buffer.from()`|**Node.js 专用**|后端开发使用，浏览器端无此 API|

### 5. 实战场景：解决 CSV/Excel 导出乱码

Excel 默认倾向于用 GBK 打开文件。如果直接下载 UTF-8 的 CSV，打开会是乱码。解决办法是加 **BOM (Byte Order Mark)**。

JavaScript

```
const str = "姓名,年龄\n张三,18";
const encoder = new TextEncoder();
const data = encoder.encode(str);

// \uFEFF 是 BOM 头，转为二进制是 [0xEF, 0xBB, 0xBF]
// 拼接 BOM + 数据
const blob = new Blob([new Uint8Array([0xEF, 0xBB, 0xBF]), data], { 
  type: 'text/csv' 
});
// ...后续调用 URL.createObjectURL(blob) 下载
```

## 五、 关键开发结论

1. **只传输，不查看**：如果只是把文件传给后端（如 FormData 上传），直接用 **File** 或 **Blob** 即可，浏览器底层会自动处理，不需要转成 ArrayBuffer，这样性能最好。
    
2. **要修改，要计算**：如果你需要计算文件的 MD5、解析 Exif 信息、或者在 Worker 中处理数据，必须转化为 **ArrayBuffer** 并套上 **Uint8Array** 或 **DataView**。
    
3. **大文件处理**：利用 `Blob.slice()` 进行物理切片，配合 Worker 线程和 `FileReader` 逐块读取，避免一次性读入内存导致浏览器崩溃。
