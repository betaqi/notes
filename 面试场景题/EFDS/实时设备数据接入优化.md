**背景问题：** 站点监控页面需要同时展示几十到上百个设备点位的实时数据，控制器通过 ZMQ 以秒级频率推送。早期实现是主线程直接处理 ZMQ 消息——接收、JSON 解析、点位映射、触发组件更新全在一个线程里。点位一多，主线程被消息处理占满，页面交互明显卡顿，拖拽拓扑图、点击设备详情都有延迟。

**方案拆分：**

我把整个数据链路分成三层：

1. **Worker 层（重计算）**：ZMQ 连接管理、Uint8Array 解码、JSON 解析、心跳检测全部在 Worker 里。Worker 收到原始消息后，做完解析和数据规整，再 postMessage 传回主线程。这样主线程完全不碰原始消息处理。
2. **分发层（轻调度）**：主线程收到 Worker 的消息后，通过 mitt 事件总线按 topic 分发。不是广播给所有组件，而是组件主动 addListener 订阅自己关心的数据，销毁时 removeListener。这样只有当前可见的组件才会收到更新。
3. **渲染层（节流控制）**：组件侧不是每收到一条消息就重渲染，而是用节流（throttle）控制更新频率。比如图表组件 200ms 更新一次，数值组件 500ms 更新一次。对于同一个点位在节流窗口内的多次推送，只取最新值。

**效果：** 优化后主线程的长任务（Long Task）基本消除，页面交互恢复流畅。用 Chrome DevTools Performance 面板观察，主线程空闲率从之前的不到 50% 提升到 80% 以上。

### Q1:  Worker 和主线程之间的通信协议怎么设计的？ 
用两组 CMD 枚举：

- `WorkerCMD`（主线程→Worker）：CONNECT、DISCONNECT、SUBSCRIBE、UNSUBSCRIBE、PUBLISH
- `MainCMD`（Worker→主线程）：DATA（点位数据）、STATUS（连接状态变化）、ERROR

每条 postMessage 的结构是 `{ cmd: MainCMD, topic: string, data: any }`。主线程收到后根据 cmd 走不同处理分支。

### Q2  postMessage 本身有序列化开销，数据量大时怎么办？
两个层面：

1. Worker 侧做数据精简——只传变化的点位，不是每次都传全量。比如一个设备有 50 个点位，这次推送只有 3 个值变了，就只传这 3 个。
2. 如果未来数据量再大，可以用 Transferable Objects（ArrayBuffer 转移所有权，零拷贝）。当前场景 JSON 序列化的开销还在可接受范围内，没有过度优化。

### Q4 有没有考虑过 SharedWorker？ 

> 考虑过。SharedWorker 可以让多个 Tab 共享一个 ZMQ 连接，减少设备端的连接压力。但我们的场景是运维人员通常只开一个 Tab 操作，多 Tab 需求不强，加上 SharedWorker 的调试体验比较差，所以没有引入。如果未来有多 Tab 需求会考虑。
