### Q1：字段映射具体怎么做的？举个例子
> **第一种是字段重命名和类型转换。** 比如拓扑模块从 PG 查出来的节点数据，数据库字段是 `node_id`、`datasource`，但前端需要的是 `id` 和 `dataSource`（camelCase），坐标字段 `x`、`y` 在数据库里是字符串，前端需要 Number 类型。我们在数据层做 map 转换，把 `node_id` 删掉赋给 `id`，`datasource` 改成 `dataSource`，坐标用 `Number()` 转型。
> 
> **第二种是字段补充。** 比如 Forerunner 返回的监控数据里只有 `appsysid`，但前端需要显示应用系统名称，我们在 BFF 层通过一个 `handleResult` 函数，根据 `appsysid` 查找对应的 `appsysname` 补充进去。
> 
> **第三种是结构性转换。** 比如 Collector 返回的 agent 数据是一个扁平数组，每个元素是 `appid:agentid` 格式的字符串，前端需要按应用分组的 map 结构，我们在 BFF 层做 split 和重组，输出 `agent_map`、`app_map` 两个索引结构。
> 
> 没有用统一的映射配置文件，是在每个模块的数据层函数里就地处理的。

### Q3：BFF 层有什么缺点？什么时候不适合用 BFF？
> BFF 的缺点我在项目里有实际感受：
> 
> **第一，多了一跳网络延迟。** 前端请求先到 BFF，BFF 再调后端服务，链路变长了。我们项目里有些接口要同时调 Forerunner 和 ES 做数据聚合，响应时间会叠加。
> 
> **第二，BFF 容易变成"胖中间层"。** 我们的 apiv1 路由文件有 1500 多行、300 多个端点，业务逻辑越堆越多，本来只是做数据转换的层，慢慢承担了越来越多的业务判断。
> 
> **第三，维护成本。** 后端服务改了字段，BFF 要跟着改映射；前端加了新需求，BFF 要加新接口。两头都要动。
> 
> **不适合用 BFF 的场景：** 如果后端服务本身就是面向前端设计的（比如 GraphQL），或者前端只有一个客户端、数据源也单一，加 BFF 就是多余的复杂度。我们项目适合用是因为底层有 9 个异构数据源，前端不可能直接对接。

### Q4：分页怎么做的？ES 的 scroll 和 PG 的 offset 怎么统一？
> 实际上我们项目里 ES 和 PG 的分页方式是统一的——都用 `from + size` 模式，没有用 ES 的 scroll API。
> 
> **具体实现：** 前端传 `page` 和 `pageSize`，路由层用 Joi 做参数校验，然后传到数据层。ES 的查询函数签名是 `query(index, query, from, size, sort, order)`，直接用 ES 的 `from/size` 参数。PG 这边用 Sequelize 的 `offset` 和 `limit`，在 `pgsqlHelper.find()` 里设置 `condition.offset = from`，`condition.limit = size`。
> 
> 两边返回格式也统一了：都是 `{ cnt, data }` 结构，`cnt` 是总数，`data` 是当前页数据。ES 从 `hits.total.value` 取总数，PG 用 Sequelize 的 `findAndCountAll`。
> 
> **为什么没用 scroll：** 我们的场景主要是前端表格分页，数据量不会特别深（一般不超过几千条），`from/size` 够用。ES 的 scroll 适合需要遍历全量数据的场景，比如导出，但会占用服务端资源（维护 scroll context），不适合高并发的分页查询。
> 
> **深分页问题：** 如果真的需要查很深的页（比如第 10000 页），ES 的 `from/size` 会有性能问题，因为 ES 要在每个分片上取 `from + size` 条数据再合并排序。解决方案是用 `search_after`，但我们项目里没遇到这个场景。

### Q6：JWT 和 Session 的区别？你们为什么两个都用？
**【发散问题｜来源：鉴权方案】【频率：🔴 高频】**
**回答框架（1-1.5 分钟）：**
> **区别：** JWT 是无状态的，token 里自带用户信息，服务端不需要存储；Session 是有状态的，服务端要维护 session 数据，客户端只存一个 session ID。
> 
> **我们为什么两个都用：** 是为了支持不同类型的客户端。
> 
> 浏览器端走 Session——用户登录后服务端创建 session，存在 ES 或内存里，cookie 名叫 `PMOne`，设了 httpOnly 和 secure。浏览器每次请求自动带 cookie，服务端从 session 里取用户信息。
> 
> API 客户端走 JWT——比如移动端或者外部系统集成，通过 `auth` header 传 token。JWT 有效期 1 小时，签发时会在数据库里存一条记录，验证时既校验签名也查数据库记录是否存在，这样可以支持主动踢人（删掉数据库记录 token 就失效了）。
> 
> **中间件的逻辑是：** 先检查请求头有没有 JWT，有就验证 JWT；没有就降级到 Session。JWT 优先级更高。这样浏览器用户和 API 用户可以共用同一套接口。
> 
> **一个细节：** 我们的 JWT 不是纯无状态的，验证时会查数据库，这是为了支持管理员强制下线功能。代价是每次请求多一次 DB 查询，但换来了对 token 生命周期的控制。

### Q7：Elasticsearch 的查询 DSL 你熟悉吗？聚合查询怎么写？
> 项目里用到的 ES 查询主要是文档检索类的，我比较熟悉的是 bool query 和基础查询。
> 
> **项目里实际用到的查询类型：**
> 
> - `query_string`：拼条件字符串，比如 `'tenant:xxx AND account.keyword:yyy'`
> - `match` 和 `match_phrase`：精确匹配字段值
> - `bool.must`：组合多个条件，比如同时匹配 name 和 folder
> - `match_all`：查全量数据（配合分页）
> 
> **聚合查询（aggs）的写法：** 项目里实际没有大量使用 ES 的聚合，但我了解基本写法：
> ```
> {
	  "aggs": {
	    "group_by_status": {
	      "terms": { "field": "status.keyword" },
	      "aggs": {
	        "avg_response_time": {
	          "avg": { "field": "responseTime" }
	        }
	      }
	    }
	  }
	}
> ```
> 常用的聚合类型有 `terms`（分组）、`avg/sum/max/min`（指标计算）、`date_histogram`（按时间分桶）、`nested`（嵌套聚合）。
> 
> **项目里分组统计更多是在 PG 侧用 SQL 的 GROUP BY 做的，** ES 主要承担的是时序数据的检索和简单过滤。

**诚实边界：** 如果面试官深入追问 ES 聚合的高级用法（pipeline aggs、composite aggs），可以说"项目里这块用得不多，我了解基本概念但没有深度实践"。

### Q8：为什么有两个数据库 ES 和 PG？它们作用分别是什么？
> 两个数据库在项目里是**互斥部署**的，不是同时使用。通过配置项 `pgsql.enable` 控制，启动时选择走 ES 还是 PG 作为主存储。
> 
> **为什么会有两套实现：** 这是一个架构演进的结果。项目早期用 ES 做全量存储——监控数据、配置数据、用户数据都存 ES。后来发现 ES 不适合存结构化的配置和关系型数据（事务支持弱、更新成本高），就引入了 PG，把配置、用户、权限这些迁移到 PG。目前默认配置是 `pgsql.enable: true`，ES 那套作为兼容保留。
> 
> **两者的定位：**
> 
> - **PG（当前主力）：** 存用户、租户、权限组、仪表盘配置、报表配置、拓扑节点等结构化数据，用 Sequelize ORM 操作，有 47 个 model
> - **ES（历史 / 可选）：** 同样的数据用 ES index 存储，查询用 DSL
> 
> **代码层面：** 两套实现都在 `src/data/backend/` 下，ES 有 21 个子模块，PG 有 27 个子模块，对外暴露相同的接口。`dbInterface.js` 根据配置决定 require 哪个实现，业务层不感知底层是哪个存储。


### Q9 后端数据源 分别是 干什么的Elasticsearch、PostgreSQL、Forerunner（APM 采集器）、NPM（网络性能服务）、Collector（数据采集服务）、SSO、Syslog
基于项目代码和架构，各数据源的职责如下：

| 数据源               | 干什么的                                     | 项目里怎么用                                                           |
| ----------------- | ---------------------------------------- | ---------------------------------------------------------------- |
| **Elasticsearch** | 分布式搜索和分析引擎，擅长全文检索和时序数据                   | 存监控数据、日志、Session；也作为早期的全量存储（现在逐步被 PG 替代）                         |
| **PostgreSQL**    | 关系型数据库，擅长结构化数据和事务                        | 存用户、租户、权限组、仪表盘配置、报表配置、拓扑节点等（47 个 model），当前主力存储                   |
| **Forerunner**    | PMOne 自研的 APM 采集器服务                      | 提供应用性能数据——Agent 列表、交易追踪、响应时间、错误率等实时监控指标，BFF 通过 HTTP 调它的 REST API |
| **NPM**           | Network Performance Monitoring，网络性能监控服务  | 提供网络层数据——网络拓扑、链路质量、丢包率、延迟等，独立部署的服务                               |
| **Collector**     | 数据采集管理服务                                 | 管理采集 Agent 的注册、状态、和应用的绑定关系（哪个 agent 采集哪个应用）                      |
| **SSO**           | 单点登录服务（支持 CAS / P13 / TAC / XXL-SSO 多协议） | 对接企业统一认证，用户登录时跳转到 SSO 验证身份，拿到 token 后回调 Fusion                   |
| **Syslog**        | 系统日志服务                                   | 接收和查询系统级日志（非应用日志），通常是网络设备、操作系统产生的日志                              |

**面试时怎么讲（30 秒版）：**

> 项目底层数据源按职责分三类：**存储类**是 ES 和 PG，PG 存配置和关系数据，ES 存时序监控数据；**采集类**是 Forerunner 和 NPM，一个管应用性能一个管网络性能，Collector 管理采集 Agent 本身的状态；**认证类**是 SSO，对接企业统一登录。BFF 层把这些异构服务统一封装给前端。

### Q10:  什么业务api 走的Forerunner
> Forerunner 是 APM 采集器，所有**运行时产生的监控数据**都从它查——应用性能、交易追踪、崩溃、告警、用户行为、AIOps 分析。而 PG/ES 存的是**配置和管理数据**——用户、权限、仪表盘配置、报表模板。

**区分口诀：**
- 数据是**实时采集产生的** → 走 Forerunner
- 数据是**用户手动配置或系统管理的** → 走 PG/
#### 和其他数据源的对比

| 数据源            | 数据特征           | 举例                  |
| -------------- | -------------- | ------------------- |
| **Forerunner** | 实时采集的性能指标、链路数据 | 响应时间、错误率、调用链、崩溃堆栈   |
| **NPM**        | 网络层采集的性能数据     | 网络拓扑、丢包率、链路延迟       |
| **PG / ES**    | 持久化的配置和关系数据    | 用户账号、权限组、仪表盘布局、报表配置 |
| **Collector**  | Agent 自身的注册和状态 | 哪些 agent 在线、绑定了哪个应用 |
