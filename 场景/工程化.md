## 一、Vite 为什么比 Webpack 快

**面试官问法**
> Vite 为什么比 Webpack 快？说说你的理解。

**口述回答**

核心原因是两者的开发模式完全不同。

Webpack 是打包驱动的，启动时要把所有模块从入口开始分析依赖，全部打包成 bundle
之后才能启动开发服务器。项目越大，依赖越多，启动就越慢，我们项目冷启动将近两分钟就是这个原因。

Vite 是按需驱动的，启动时不打包，直接启动开发服务器。浏览器请求哪个文件，Vite 才去处理那个文件，利用浏览器原生的
ESM 支持，按需编译按需加载。所以无论项目多大，启动时间基本是固定的，非常快。

热更新方面，Webpack 改一个文件要重新打包整个模块链，项目大了热更新也会很慢。Vite
改一个文件只需要让浏览器重新请求这一个模块，其他模块的缓存完全不受影响，所以热更新是毫秒级的。

**加分补充**

Vite 还做了一个预构建的机制，专门处理第三方依赖。因为 node_modules 里很多库是 CommonJS 格式，浏览器不认识，Vite
在第一次启动时用 esbuild 把这些依赖预构建成 ESM 格式并缓存起来，后续启动直接用缓存，不需要重复处理。esbuild
是用 Go 写的，比 JavaScript 写的构建工具快 10 到 100 倍，这也是 Vite 快的原因之一。

## 二、Tree Shaking 是什么

**面试官问法**
> Tree Shaking 是什么？Vite 和 Webpack 分别怎么实现的？

**口述回答**

Tree Shaking 是一种打包优化手段，目的是把代码里没有用到的部分在打包时自动剔除掉，减少最终产物的体积。名字来源于摇树，把枯叶摇掉，只留下有用的部分。

实现 Tree Shaking 的前提是代码必须用 ESM 格式，也就是 import/export，因为 ESM
是静态的，打包工具在编译阶段就能分析出哪些导出被用到了，哪些没被用到。CommonJS 的 require
是动态的，运行时才知道导入了什么，没办法做静态分析。

Webpack 的 Tree Shaking 是基于静态分析，标记没有被引用的 export，配合 TerserPlugin 在压缩阶段把这些死代码删掉。需要在
package.json 里配置 sideEffects 字段，告诉 Webpack 哪些文件有副作用不能删，哪些可以安全剔除。

Vite 生产环境用 Rollup 打包，Rollup 是最早支持 Tree Shaking 的打包工具，对 ESM 的支持和静态分析能力比
Webpack 更成熟，Tree Shaking 效果通常更好。

**加分补充**

实际项目里 Tree Shaking 失效的情况很常见。最常见的原因是用了有副作用的写法，比如在模块顶层执行了函数调用，或者用了
CommonJS 的库。另外 Vue、lodash 这类库如果是整体导入而不是按需导入，Tree Shaking 也不会生效，需要改成具名导入的方式。

## 三、ESLint 规范怎么制定的

**面试官问法**
> 你们的 ESLint 规范是怎么制定的？有没有自定义规则？

**口述回答**

我们的规范是基于已有的成熟方案扩展的，没有从零写规则。

基础用的是 eslint-config-airbnb 或者 @antfu/eslint-config 这类社区成熟方案，再结合项目实际情况做调整。比如我们用
Vue3 + TypeScript，会加上 eslint-plugin-vue 和 @typescript-eslint 相关规则，对 Vue 单文件组件和
TypeScript 类型做额外约束。

Prettier 负责代码格式，ESLint 负责代码质量，两者分工明确，通过 eslint-prettier 做集成，避免规则冲突。

规范落地这块，我们在 package.json 里配置了 lint-staged 配合 husky，在 git commit 的时候自动对暂存区的文件跑
ESLint 检查，不通过就拒绝提交，保证入库代码的质量。

**加分补充**

自定义规则这块我们有做过一些项目级的约定，不是写 AST 插件，而是通过 ESLint 的 no-restricted-syntax
规则限制一些我们不想用的写法。比如限制直接操作 DOM、限制某些全局变量的使用等，用配置的方式就能实现，不需要写自定义插件。

## 四、Vite 迁移踩过的坑

**面试官问法**
> 迁移过程中遇到最大的坑是什么？怎么解决的？

**口述回答**

迁移过程中踩了三个比较典型的坑。

第一个是 CommonJS 和 ESM 的兼容问题。项目里有几个老库只有 CommonJS 格式，Vite 开发环境下直接报错。解决方法是在
optimizeDeps.include 里把这些库显式声明，让 Vite 预构建阶段提前把它们转成 ESM，之后就正常了。

第二个是动态导入路径的问题。Webpack 里有些写法是用变量拼接路径做动态导入，Vite
对这种写法支持不好，会报警告甚至报错。我们把这类写法改成了 import.meta.glob 来批量导入，或者改成显式的静态路径。

第三个是环境变量的问题。Webpack 里用的是 process.env.VUE_APP_XXX，Vite 里改成了
import.meta.env.VITE_XXX，变量命名前缀也从 VUE_APP 改成了 VITE，项目里用到环境变量的地方都要批量替换，比较繁琐但不复杂。

**加分补充**

还有一个容易忽略的点是 require 的用法。Vite 不支持 require，但老项目里经常有 require 动态加载图片或者
JSON 的写法，需要全部改成 import 或者 new URL 的方式。这类问题不会在启动时报错，往往是运行到具体功能才发现，所以迁移后要做比较全面的功能回归测试。

## 五、开发没问题生产报错

**面试官问法**
> Vite 开发环境是 ESM，生产环境是 Rollup 打包，有没有遇到过开发没问题但生产报错的情况？

**口述回答**

遇到过，这是 Vite 迁移里比较经典的问题。

开发环境和生产环境的构建机制完全不同，开发环境是浏览器直接加载 ESM 文件，生产环境是 Rollup 打包成
bundle，两者对代码的处理方式有差异，所以有些问题只在生产环境才会暴露。

我们遇到过的一个典型情况是某个第三方库在开发环境下通过预构建可以正常使用，但生产环境 Rollup
打包时处理这个库的方式不一样，导致某些方法找不到。解决方式是在 vite.config.ts 里把这个库加到
build.commonjsOptions.include 里，让 Rollup 正确处理它的 CommonJS 格式。

**加分补充**

这个问题给我的教训是，Vite 迁移完成后不能只验证开发环境，一定要同步跑一次生产构建并做完整的功能回归，尤其是一些边缘功能和第三方库的使用场景。我们后来在
CI 流程里加了生产构建的验证步骤，每次提交都会跑一次 vite build，确保生产构建不会挂。

## 六、manualChunks 拆分策略

**面试官问法**
> 你提到用 manualChunks 拆分 Vendor，拆分的策略是什么？按什么维度拆的？

**口述回答**

拆分的核心目标是利用浏览器缓存，让变动频繁的代码和变动不频繁的代码分开打包。

我们按三个维度拆分。

第一类是框架核心，把 Vue、Vue Router、Pinia 这些基础框架单独打成一个
chunk，这些库版本很稳定，基本不会变动，用户第一次加载后就长期缓存。

第二类是可视化库，把 ECharts、G6、X6 这类体积比较大的可视化库单独打包，因为这些库体积动辄几百
KB，单独拆出来可以避免因为业务代码变动导致这个大文件缓存失效。

第三类是业务代码，按页面或者模块做代码分割，配合路由懒加载，用户访问哪个页面才加载对应的
chunk，减少首屏加载体积。

**加分补充**

拆分也不是越细越好，chunk 太多会增加 HTTP 请求数，在 HTTP/1.1 下反而有副作用。我们的原则是单个 chunk
控制在 200KB 到 500KB 之间，太大的拆开，太小的合并，用 rollup-plugin-visualizer 做打包分析，可视化地看每个
chunk 的构成和大小，根据数据调整拆分策略。

## 复习重点提示

- **Vite 快的核心** — 按需编译 + 原生 ESM，不是打包驱动，记住这个本质
- **Tree Shaking 前提** — 必须是 ESM 静态导入，CommonJS 不支持
- **迁移三个坑** — CommonJS 兼容、动态路径、环境变量，每个都能展开说
- **撤销重做** — 命令模式 + 快照，说清楚 execute/undo 和历史栈结构
- **内存泄漏定位** — Memory 面板 + 堆快照对比，这个方法论很重要
- **数字要记住** — 冷启动两分钟 → 30 秒，热更新几秒 → 毫秒级
