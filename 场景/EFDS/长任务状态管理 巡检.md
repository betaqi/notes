**背景问题：** 固件升级和设备巡检是典型的长耗时任务——固件升级从上传到安装完成可能要 3-5 分钟，巡检一轮可能要 1-2 分钟。这期间用户可能刷新页面、切换路由、网络可能抖动、设备回执可能延迟。如果前端只靠内存状态，页面一刷新就丢了，用户不知道任务到底成功没有。

**我的方案是设计一套任务状态流转管理：**
1. **状态定义**：每个长任务有明确的状态枚举——idle → pending → executing → success/failed/cancelled/timeout。不同任务类型的中间状态不同，比如固件升级是 idle → uploading → installing → success/failed。
2. **指令回执驱动**：状态转换不是前端自己猜的，而是由 ZMQ 回执消息驱动。发出指令后进入 pending，收到 `result: 'progress'` 进入 executing，收到 `result: 'success'` 进入 success。每条回执消息有唯一 id 和发出的指令对应。
3. **超时检查**：如果在预期时间内没收到回执（比如升级 5 分钟没响应），主动轮询后端状态接口（`getUpgradeStatus`）。如果后端也没有结果，标记为 timeout 并提示用户。
4. **页面恢复重查**：页面刷新或重新进入时，先查询后端当前任务状态，恢复到正确的 UI 状态。比如用户刷新页面后，发现后端返回"升级中"，前端直接进入 installing 状态继续展示进度。

**效果：** 解决了三个问题——页面刷新不丢状态、网络抖动不会假死、回执延迟不会卡在中间态。

### 用 Generator 管理长任务状态
> 我用 Generator 来编排长任务的状态流转。好处是把异步的、事件驱动的流程写成同步风格的线性代码——每个 yield 就是一个等待外部事件的挂起点。外部收到 ZMQ 回执时调用 next 推进流程，超时时调用 return 终止。相比用一堆 if/switch 判断当前状态再决定下一步，Generator 的流程意图更清晰，也更容易加新的中间状态。



> 巡检模块是一个多步骤的长流程——RS485、DIO、CAN、LED 四个模块按顺序巡检，每个模块下面有多个子项。我的实现核心是**消息 ID 追踪 + 回执驱动推进**：
> 
> 每次发出巡检指令时，生成唯一 ID 注册到一个 Map 里，记录这条消息属于哪个步骤、什么动作。设备端执行完一个子项就推送一条回执，前端通过 ID 反查是哪个步骤的结果，更新对应子项的状态。当一个步骤的所有子项都完成后，自动发起下一个步骤的指令。
> 
> 容错方面做了三件事：
> 
> 1. **超时**：Worker 层设置 7 秒超时，超时后清除状态通知用户重试
> 2. **页面离开保护**：路由守卫拦截离开，发送 testStop 指令让设备端停止当前巡检
> 3. **设备锁**：每 4 秒发一次 lock 心跳，保持独占访问，防止多人同时操作
