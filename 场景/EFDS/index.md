### Q1: 这个项目是干什么的？
> 这是一个部署在储能电站边缘侧的设备运维管理平台。管理的设备包括 BMS（电池管理系统）、PCS（储能变流器）、电表、消防温控这些。我做的是前端部分，核心解决两个问题：一是让运维人员实时看到设备状态和数据，二是能远程下发控制指令，比如充放电策略配置、固件升级、巡检任务。技术上通过 ZMQ WebSocket 实现秒级的双向通信。

关键词记住：**储能电站 → 边缘设备 → 运维管理 → 实时监控 + 远程控制**

### Q2:    负责什么
> 我负责三块：
> 
> 第一块是核心业务模块开发——设备拓扑（X6 树形布局 + 实时状态着色）、EMS 策略调控（充放电时段配置 + ZMQ 下发）、固件升级（分片上传 + 状态轮询）、[[长任务状态管理 巡检]]、SSH 远程终端（xterm.js）。巡检（RS485/CAN/DIO/LED 四种接口）

> 第二块是实时消息接入层——我封装了一套 Web Worker 里跑的 ZMQ 客户端，处理主题订阅、心跳检测、消息解析和分发。这是整个系统实时性的基础，所有需要实时数据的页面都依赖这一层。
> 
> 第三块是参与了一个低代码大屏编辑器的二次开发，基于 GoView，扩展储能行业组件和 MQTT 点位绑定。

### Q3: 为什么选 ZMQ 而不是纯 WebSocket 或 MQTT
**你的讲法：**
> 三个原因：
> 
> **一、场景匹配。** 我们是边缘侧部署，前端直连控制器，不经过云端。ZMQ 是无 broker 的点对点通信，不需要额外部署消息中间件，架构更轻。
> 
> **二、消息模式丰富。** ZMQ 原生支持 Pub/Sub（实时数据订阅）和 Req/Rep（指令下发等回复），一套协议覆盖两种场景。纯 WebSocket 需要自己实现消息路由和主题过滤。
> 
> **三、延迟更低。** 没有 broker 中转，消息从控制器直达前端，适合秒级甚至亚秒级的设备数据推送。
> 
> MQTT 我们也用了，但是在大屏场景——多个站点的数据汇聚到云端时走 MQTT，因为那个场景需要 broker 做消息路由和持久化。

### Q4:  Worker 代码细节 + EMS 策略调控

> Worker 内部结构：
> 
> 1. **CMD 枚举通信协议**：主线程和 Worker 之间用 `WorkerCMD`（主线程→Worker）和 `MainCMD`（Worker→主线程）两组枚举定义消息类型，比如 CONNECT、DISCONNECT、SUBSCRIBE、PUBLISH、STATUS。
> 2. **连接管理**：Worker 内维护 subClient（订阅）和 pubClient（发布）两个 ZMQ 客户端实例，连接到 `ws://{host}:15555` 和 `ws://{host}:15556`。
> 3. **心跳检测**：订阅 `HEARTBEAT` 主题，每收到一次心跳更新 `lastHeartbeatTime`。另起一个 1 秒间隔的定时器检查，如果超过 3 秒没收到心跳，postMessage 通知主线程连接异常。
> 4. **消息解析**：收到 Uint8Array → TextDecoder 解码 → JSON.parse → postMessage 传回主线程。
> 5. **主线程分发**：主线程收到 postMessage 后，通过 mitt 事件总线 emit 到 `zmqData` 或 `zmqCmd` 频道，各组件通过 addListener/removeListener 订阅。

#### EMS 策略调控
**你的讲法：**
> EMS 策略调控的业务含义是：运维人员配置储能设备在一天中不同时段的充放电策略。比如谷电时段充电、峰电时段放电，实现削峰填谷。
> 
> 前端实现：
> 
> - 用 ECharts 折线图展示 24 小时的功率曲线，直观看到各时段的充放电分布
> - 弹窗编辑器配置每个时段的起止时间和功率参数（有时间冲突校验）
> - 配置完成后，通过 ZMQ Pub 发送到控制器，消息格式是 `{id, action, reply: 'yes', params: {...}}`
> - 等待 Sub 回执确认下发成功（result: success/failure）
> 
> 点位数据结构里有 `viewType` 字段区分展示方式：`time-start`、`time-end` 用时间选择器，`input` 用数字输入框，`select` 用下拉框。每个点位有 `permission: 'RO'|'RW'` 控制是否可编辑。

### Q5:   固件升级分片上传
> 流程是：选择文件 → 前端分片 → 逐片上传 → 上传完成 → 调用升级接口 → 轮询升级状态。
> 
> 具体参数：
> 
> - **分片大小**：2MB（`1024 * 1024 * 2`）
> - **上传方式**：串行逐片上传（不是并发），每片带 offset、chunkIndex、chunkTotal、fileName、fileSize、appId
> - **进度回调**：每片上传完更新进度百分比，通过 ProgressListener 接口回调给 UI 层
> - **取消机制**：AbortController，用户点"停止"时 abort 当前请求
> - **升级状态**：上传完成后调用 `appUpgrade` 接口触发安装，然后用 setTimeout 轮询 `getUpgradeStatus`，直到返回 success 或 failed
> - **UI 状态**：上传中显示进度条（百分比），安装中显示 indeterminate 进度条（"安装中，请耐心等待"）

**如果追问"为什么串行不并发"：**  
边缘设备的处理能力和网络带宽有限，并发上传可能导致设备端缓冲区溢出或请求超时。串行虽然慢一点，但稳定性更好。

**如果追问"断点续传"：**  
当前实现没有做断点续传（abort 后重新上传是从头开始）。如果要做，需要后端支持查询已接收的 chunkIndex，前端跳过已上传的片段。
