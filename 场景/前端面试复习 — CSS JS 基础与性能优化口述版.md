## 一、Flex 和 Grid 的区别

**口述回答**

Flex 是一维布局，只能控制一个方向，要么横向要么纵向。适合做导航栏、按钮组、卡片列表这类单行或单列的排列场景，用起来简单直接。

Grid 是二维布局，可以同时控制行和列，适合做整体页面布局、表格型内容、需要精确控制行列位置的复杂布局。

简单说就是：一行或一列用 Flex，需要同时控制行列就用 Grid。

实际项目里两者经常配合用，外层页面结构用 Grid 做整体骨架，内部组件用 Flex 做排列对齐。

**加分补充**

Flex 有一个容易踩的坑是子元素设置了 flex: 1 之后在某些浏览器下宽度计算不符合预期，需要配合 min-width: 0 解决溢出问题。Grid 的兼容性在 IE11 以下支持不好，老项目要注意。

## 二、响应式布局和媒体查询

**口述回答**

响应式布局是让页面能自动适配不同屏幕尺寸，核心手段有三个。

第一是弹性单位，用百分比、vw、vh、rem 代替固定 px，让尺寸跟随屏幕变化。

第二是媒体查询，根据屏幕宽度应用不同的 CSS 规则，比如手机上单列展示，桌面端三列展示。

第三是弹性图片，图片设置 max-width: 100%，让图片在小屏幕上自动缩小不溢出。

断点设计上通常按主流设备分三档：移动端 768px 以下，平板 768px 到 1024px，桌面端 1024px 以上。具体数值根据设计稿和产品需求调整，不是固定的。

**加分补充**

媒体查询推荐移动端优先的写法，默认样式写手机样式，用 min-width 向上覆盖。这样比桌面优先用 max-width 向下覆盖的写法更合理，因为移动端流量占大头，而且移动端样式往往更简单，作为默认值代码量更少。

## 三、CSS 优先级规则

**口述回答**

CSS 优先级从高到低分四个等级。

第一级是内联样式，也就是直接写在 style 属性里的，优先级最高。

第二级是 ID 选择器，用 # 开头的。

第三级是类选择器、属性选择器、伪类，比如 .class、[type="text"]、:hover。

第四级是标签选择器和伪元素，比如 div、p、::before。

选择器可以叠加，叠加的时候每一级对应一个权重值，通常表示为 0,0,0,0 四位数，ID 贡献第二位，类贡献第三位，标签贡献第四位，比较时从高位往低位比。

!important 可以强制覆盖所有优先级，但要谨慎用，滥用会让样式难以维护。

同等优先级的情况下，后面的规则覆盖前面的。

**加分补充**

有一个容易混淆的点是伪类和伪元素的区别。伪类是 :hover、:first-child 这类描述元素状态的，伪元素是 ::before、::after 这类创建虚拟元素的，两者优先级不同，伪类是类级别，伪元素是标签级别。

## 四、BFC 是什么有什么用

**口述回答**

BFC 全称块级格式化上下文，可以理解为一个独立的渲染区域，内部的布局不会影响外部，外部也不会影响内部。

触发 BFC 的常见方式有：设置 overflow 为 hidden 或 auto、设置 float、设置 position 为 absolute 或 fixed、设置 display 为 flex 或 inline-block。

BFC 在实际开发中主要解决三个问题。

第一个是清除浮动。父元素里的子元素都浮动后，父元素高度塌陷，给父元素触发 BFC 就能让父元素包住浮动子元素。

第二个是防止 margin 合并。两个相邻的块元素 margin 会合并取最大值，把其中一个放进 BFC 容器里就能防止合并。

第三个是防止文字环绕浮动元素。给文字容器触发 BFC 可以让文字不环绕，形成两列布局。

**加分补充**

现在项目里用 Flex 和 Grid 的场景多了，直接用 overflow hidden 清除浮动的场景少了，但面试还是高频考点，说清楚三个实际用途就够了。

## 五、事件冒泡捕获和事件委托

**口述回答**

浏览器事件传播分三个阶段。第一阶段是捕获，事件从 window 往下传到目标元素。第二阶段是目标阶段，事件到达目标元素。第三阶段是冒泡，事件从目标元素往上传回 window。

addEventListener 第三个参数控制监听哪个阶段，默认 false 是冒泡阶段，true 是捕获阶段。实际开发中基本都用冒泡，捕获用得很少。

事件委托是利用冒泡机制，把子元素的事件监听器绑在父元素上，通过 event.target 判断是哪个子元素触发的。好处是不需要给每个子元素单独绑定事件，节省内存，而且动态新增的子元素也能自动响应事件，不需要重新绑定。

典型场景是列表，给列表容器绑一个 click 事件，判断 target 是哪个列表项，比给每个列表项绑事件高效很多。

**加分补充**

使用事件委托要注意 stopPropagation 的问题，如果子元素里有阻止冒泡的逻辑，事件就传不到父元素的委托监听器了。另外 target 和 currentTarget 的区别要清楚，target 是实际触发事件的元素，currentTarget 是绑定监听器的元素。

## 六、闭包是什么有什么实际用途

**口述回答**

闭包是函数加上它能访问的外部变量的组合。简单说就是一个函数记住了它定义时所在作用域里的变量，即使这个作用域已经执行完了，这些变量也不会被销毁。

最常见的例子是函数里返回一个函数，内层函数可以访问外层函数的变量，外层函数执行完之后内层函数还能继续用那些变量。

实际用途主要有三个。

第一是封装私有变量。用闭包可以创建只有特定函数能访问的私有数据，外部无法直接修改，类似面向对象的私有属性。

第二是函数工厂。根据参数生成不同配置的函数，比如生成不同倍数的乘法函数。

第三是防抖节流。防抖和节流里用闭包保存定时器 ID 和上次执行时间，每次调用都能访问到上一次的状态。

**加分补充**

闭包容易导致内存泄漏，因为闭包持有外部变量的引用，外部变量就无法被垃圾回收。在事件监听或者定时器里使用闭包，如果不手动清理，会一直占用内存。这也是我们项目里内存泄漏排查时重点关注的方向之一。

## 七、Promise 和 async/await 的区别

**口述回答**

两者本质上是同一套东西，async/await 是 Promise 的语法糖，底层还是 Promise，只是写法不一样。

Promise 用链式调用，then 处理成功，catch 处理错误，代码是横向扩展的，嵌套多了还是会有回调地狱的感觉。

async/await 让异步代码看起来像同步代码，用 try/catch 处理错误，可读性更好，更符合人的阅读习惯，是现在的主流写法。

两者的执行时机是一样的，都是微任务，在当前宏任务执行完之后、下一个宏任务开始之前执行。

**加分补充**

async/await 有一个常见的性能问题，就是串行等待。如果两个异步操作没有依赖关系，用 await 一个等一个是浪费时间的，应该用 Promise.all 并行执行，等两个都完成再继续，这样总时间是最长那个请求的时间而不是两个相加。

## 八、this 指向和箭头函数

**口述回答**

普通函数的 this 是动态的，取决于调用时的上下文，谁调用就指向谁。直接调用指向 window，对象方法调用指向这个对象，new 调用指向新创建的实例，call/apply/bind 可以手动指定。

箭头函数没有自己的 this，它的 this 是定义时继承外层作用域的 this，不会因为调用方式改变。

实际开发中最常见的场景是在 Vue 或者类里面，回调函数里用 this 访问组件实例，用普通函数 this 会丢失，改成箭头函数就能正确指向外层的 this。

**加分补充**

箭头函数除了 this 的问题，还有几个特性要记住：没有 arguments 对象、不能用 new 调用、没有 prototype 属性。面试偶尔会问到这些细节，知道就行。

## 九、页面加载慢怎么排查

**口述回答**

排查页面加载慢分三个步骤。

第一步是用 Chrome DevTools 的 Network 面板看网络请求，找出耗时最长的资源，看是 JS 文件太大、图片太大还是接口响应慢。

第二步是看 Performance 面板，录制页面加载过程，找出哪个阶段耗时最多，是解析 HTML、执行 JS 还是渲染。

第三步是看 Lighthouse，它会自动分析页面并给出优化建议，包括首屏时间、可交互时间等核心指标和具体的优化方向。

定位到问题之后，常见的优化手段是压缩资源体积、使用 CDN、开启 Gzip、图片懒加载、JS 拆包按需加载、接口合并减少请求数。

**加分补充**

有一个容易忽略的点是 render-blocking 资源，也就是阻塞渲染的资源。CSS 和 JS 如果放在 head 里同步加载，会阻塞浏览器渲染，页面白屏时间会很长。CSS 要正常放 head，但 JS 要加 defer 或 async 属性，或者放到 body 底部，避免阻塞渲染。

## 十、图片优化有哪些手段

**口述回答**

图片优化主要从四个方向入手。

第一是格式选择。照片用 JPG，透明图用 PNG，图标和简单图形用 SVG，条件允许的话用 WebP，同等质量下体积比 JPG 小 30% 左右。

第二是压缩体积。用工具对图片做有损或无损压缩，在不明显影响视觉质量的前提下减小文件大小。

第三是懒加载。页面首屏只加载可视区域内的图片，用户滚动到图片位置时再加载，减少首屏请求数和流量。现在可以直接用 img 标签的 loading="lazy" 属性，浏览器原生支持。

第四是响应式图片。用 srcset 属性根据屏幕分辨率提供不同尺寸的图片，手机上不加载桌面端的大图，节省流量。

**加分补充**

雪碧图也是一种传统优化手段，把多个小图标合并成一张大图，减少 HTTP 请求数。但现在 HTTP/2 支持多路复用，多个请求可以并行，雪碧图的优势没那么明显了，而且维护成本高，新项目一般不用。图标更推荐用 SVG 或者 Icon Font。

## 十一、什么是异步加载怎么实现

**口述回答**

异步加载是指资源不在页面初始化时一起加载，而是在需要的时候再加载，目的是减少首屏加载的资源体积，加快页面初始可见速度。

JS 的异步加载有几种方式。

第一种是 script 标签加 defer 属性，JS 文件异步下载，等 HTML 解析完之后再按顺序执行，不阻塞渲染。

第二种是 script 标签加 async 属性，JS 文件异步下载，下载完立即执行，不保证执行顺序，适合没有依赖关系的独立脚本。

第三种是动态 import，也就是代码分割，在需要某个功能的时候才动态加载对应的 JS 模块，是现代前端框架里最常用的方式。Vue 和 React 的路由懒加载就是基于动态 import 实现的。

**加分补充**

资源预加载也是一种相关手段，用 link rel="preload" 提前加载后续页面会用到的资源，利用浏览器空闲时间下载，等真正需要时直接从缓存取，减少等待时间。

## 十二、浏览器兼容性问题

**面试官问法**

> 你遇到过哪些兼容性问题，怎么解决的？

**口述回答**

遇到过几类比较典型的兼容性问题。

第一类是 CSS 样式兼容。Flex 布局在旧版 Safari 上需要加 -webkit- 前缀，某些 CSS 属性在不同浏览器下默认样式不一样，用 normalize.css 或 reset.css 统一基础样式。

第二类是 JS API 兼容。比如 Promise、fetch、IntersectionObserver 在旧浏览器里不支持，需要引入 polyfill 补齐。我们项目里用 Babel 配合 @babel/preset-env，根据目标浏览器版本自动转译语法和注入 polyfill。

第三类是事件兼容。老版本 IE 用的是 attachEvent 而不是 addEventListener，不过现在 IE 基本已经不需要支持了。

**加分补充**

处理兼容性问题的流程是：先用 Can I Use 查目标浏览器对某个特性的支持情况，再决定是加前缀、引入 polyfill 还是换备用方案。不要凭感觉猜，以数据为准。

## 十三、CSS 前缀是什么

**口述回答**

CSS 前缀是浏览器厂商在 CSS 标准正式确定之前，用来实验性支持新特性的私有前缀。

常见的前缀有四个：-webkit- 是 Chrome 和 Safari 用的，-moz- 是 Firefox 用的，-ms- 是 IE 和旧版 Edge 用的，-o- 是旧版 Opera 用的。

比如 transform 这个属性早期需要写四个版本加上无前缀版本，现在主流浏览器都支持标准写法了，一般不需要手动加了。

实际项目里不需要手动写前缀，用 PostCSS 的 autoprefixer 插件，配置好目标浏览器版本，构建时自动加上需要的前缀，省心很多。

## 十四、jQuery Ajax 和原生 fetch 的区别

**口述回答**

jQuery 的 Ajax 是对 XMLHttpRequest 的封装，提供了简洁的 API，解决了早期浏览器兼容问题，在 jQuery 时代是主流方案。

原生 fetch 是浏览器新的网络请求 API，基于 Promise，语法更现代，代码更简洁，不需要引入额外库。

主要区别有几点。

第一，fetch 默认不携带 Cookie，需要手动配置 credentials: 'include'，jQuery Ajax 默认携带。

第二，fetch 对错误的处理不太直觉，HTTP 4xx 和 5xx 错误不会触发 reject，只有网络断开才会 reject，需要手动判断 response.ok。jQuery Ajax 在服务端返回错误状态码时会触发 error 回调。

第三，fetch 原生不支持请求超时和取消，需要配合 AbortController 实现，jQuery Ajax 有内置的 timeout 参数。

现代项目里一般不直接用 fetch，而是用 axios，它基于 XMLHttpRequest，兼容性更好，功能也更完善，支持拦截器、自动转换 JSON、超时配置等。

## 十五、微信小程序和 H5 的区别

**口述回答**

微信小程序和 H5 的核心区别是运行环境不同。

H5 运行在浏览器里，用标准的 HTML、CSS、JavaScript，可以在任何浏览器打开。

小程序运行在微信客户端的双线程环境里，逻辑层和渲染层分离，逻辑层跑 JS，渲染层用 WXML 和 WXSS，两者通过微信客户端通信，不能直接操作 DOM。

能力上的区别是小程序可以调用微信原生能力，比如扫码、支付、获取用户信息、蓝牙、地理位置等，H5 通过 JSSDK 可以调用部分，但权限和体验比小程序差。

性能上小程序通常比 H5 流畅，因为有微信客户端的原生渲染加持，不受浏览器限制。

开发上小程序有自己的语法和组件体系，WXML 类似 HTML，WXSS 类似 CSS，但有限制，比如不支持操作 DOM、不支持某些 CSS 选择器。

**加分补充**

现在很多团队会用 uni-app 或者 Taro 做跨端开发，一套代码同时生成 H5 和小程序，减少重复开发成本。如果项目同时有 H5 和小程序需求，跨端框架是比较主流的选择。

## 复习重点提示

- **Flex vs Grid** — 一维用 Flex，二维用 Grid，实际项目配合用
- **BFC 三个用途** — 清除浮动、防止 margin 合并、防止文字环绕，记住这三个
- **事件委托** — 绑在父元素，用 target 判断子元素，节省内存且支持动态元素
- **闭包内存泄漏** — 事件监听和定时器里用闭包要记得清理
- **async/await 并行** — 无依赖的异步操作用 Promise.all，不要串行 await
- **图片优化四个方向** — 格式、压缩、懒加载、响应式，说全了
- **fetch 两个坑** — 不携带 Cookie、4xx/5xx 不 reject，这两点很容易考
- **小程序 vs H5** — 运行环境、原生能力、双线程架构是核心差异
